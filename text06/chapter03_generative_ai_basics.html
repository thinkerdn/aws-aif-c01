3-1 生成AIの基本

生成AIの概念とアーキテクチャ

生成AI（Generative AI）の定義

生成AIとは、学習データのパターンを学習し、テキスト・画像・コード・音声などの新しいコンテンツを自律的に生成できるAIのことです。単なる分類や予測にとどまらず、存在しなかったコンテンツを創り出す能力を持ちます。

識別AI vs 生成AIの違い

観点 識別AI (Discriminative AI) 生成AI (Generative AI)
目的 入力データを分類・予測 新しいコンテンツを生成
出力 ラベル・スコア・カテゴリ テキスト・画像・コード・音声
主な用途 スパム検出、画像分類、感情分析 文章生成、画像生成、コード補完
学習 入力→正解ラベルのマッピング データ分布の学習と新サンプル生成
代表例 Amazon Rekognition, Comprehend Claude, GPT-4, DALL-E, Titan

主要なアーキテクチャ

アーキテクチャ 仕組み 主な用途 代表例
Transformer 自己注意機構（Self-Attention）により文脈全体を並列処理。LLMの基礎技術。 テキスト生成、翻訳、コード生成 BERT, GPT, Claude, Llama
GAN
（敵対的生成ネットワーク）		 生成器（Generator）と識別器（Discriminator）が対戦学習。生成器はリアルなデータを生成し、識別器は本物と偽物を区別する。 リアルな画像・動画生成、データ拡張 StyleGAN, BigGAN
VAE
（変分オートエンコーダー）		 入力を潜在空間（Latent Space）に圧縮・再構成。潜在空間をサンプリングして新データを生成。 画像生成、異常検知、データ補完 β-VAE, VQ-VAE
拡散モデル
（Diffusion Models）		 データに徐々にノイズを加え（順拡散）、そのノイズを逆向きに除去する（逆拡散）プロセスを学習。高品質な画像生成を実現。 高品質画像・動画生成 Stable Diffusion, DALL-E 3, Titan Image

各処理ステップの詳細

ステップ 説明 試験ポイント
トークン化 テキストを数値トークンに分割。単語、サブワード、文字などの単位で分割される。 日本語は英語より多くのトークンを消費する傾向がある。
埋め込み（Embedding） 各トークンを高次元ベクトル空間に配置。意味的に近い単語は近い位置に配置される。 Amazon Titan Embeddings は RAG での類似検索に使用される。
Self-Attention 入力シーケンス内の全トークン間の関係性を並列計算。長距離依存関係を捉える。 Transformerの中核技術。コンテキストウィンドウの処理に関係。
次トークン予測 これまでのトークンから次のトークンを確率的に予測。温度パラメータ（Temperature）で多様性を制御。 Temperature=0で最確率、Temperature=1で多様性が高まる。
パラメータ規模 数十億（Billion）〜数兆（Trillion）のパラメータを保持。規模が大きいほど一般的に性能が高い。 大きいモデル＝高コスト・高性能。小さいモデル＝低コスト・高速。

生成AIのユースケース

コンテンツ生成

記事・ブログ執筆、マーケティングコピー、製品説明文の自動生成など、文章コンテンツの大量生産を効率化します。人間のライターが行っていた作業を補助・代替し、コンテンツ生産速度を大幅に向上させます。

AWSサービス

Amazon Bedrock + Claude モデル / Amazon Titan Text Generator

コード生成・支援

コードの自動補完、バグ修正、コードレビュー、テストコード生成など、開発者の生産性を向上させます。IDEプラグインとして開発ワークフローに統合できます。

AWSサービス

Amazon Q Developer（旧 Amazon CodeWhisperer）— IDE（VS Code, JetBrains）プラグインとして利用可能。Bedrock + Claude でのコード生成。

質問応答・ナレッジ検索（RAG）

社内ドキュメント、製品マニュアル、FAQに基づいたカスタマーサポートbot。RAG（Retrieval-Augmented Generation）により、最新かつ社内固有の情報を根拠とした回答を生成します。

AWSサービス

Amazon Q Business（企業データのチャット型検索）/ Amazon Bedrock Knowledge Bases（マネージドRAG）/ Amazon Kendra（インテリジェント検索）

要約・情報抽出

長文ドキュメントの要約、会議議事録の自動生成、契約書から重要条項の抽出など、情報処理の効率化に活用されます。

AWSサービス

Amazon Bedrock + Claude（長いコンテキストウィンドウにより長文処理が得意）

画像生成・編集

マーケティング画像・バナーの自動生成、製品プロトタイプのビジュアル化、eコマースの商品画像作成などに活用されます。

AWSサービス

Amazon Titan Image Generator（Bedrockで利用）/ Stability AI（Stable Diffusion）on Bedrock

翻訳・多言語対応

ドキュメント翻訳、リアルタイム多言語チャット対応、グローバルコンテンツのローカライズなどに活用されます。

AWSサービス

Amazon Translate（専用翻訳サービス）/ Amazon Bedrock（多言語LLMによる高品質翻訳）

基盤モデル（FM）のライフサイクル

事前学習（Pre-training）
インターネット上の大量のテキストデータを用いた自己教師あり学習（次トークン予測）。膨大なGPUリソースと時間が必要。基盤となる言語知識・世界知識を獲得する。
ファインチューニング（Fine-tuning）
特定タスクや業界固有のデータで追加学習。事前学習モデルのパラメータを特化データで微調整し、特定用途での精度を向上させる。AWSではBedrock Fine-tuningで実施可能。
RLHF（人間のフィードバックによる強化学習）
人間のアノテーターが複数の回答を評価・ランキングし、その評価を用いて報酬モデルを構築。PPO（近似方策最適化）で出力を人間の好みに合わせ、安全性と有用性を向上させる。
デプロイ（Deploy）
APIエンドポイントとして公開。スケーラブルなインフラで本番運用。AWSではBedrockのオンデマンドまたはプロビジョンドスループットで利用可能。
モニタリング（Monitoring）
性能劣化（モデルドリフト）の検知、有害出力・バイアスの監視、コストとレイテンシの追跡。Amazon CloudWatch、Bedrock Model Evaluation を活用。
更新・廃止（Update / Deprecation）
新バージョンへの移行、モデルの廃止対応。AWSはモデル廃止の6ヶ月前に通知するポリシーを採用している。
事前学習は大量データ＋大量計算リソースが必要で費用が非常に高額。企業の用途にはファインチューニングや RAG による適応が現実的。Bedrockはモデルの事前学習を不要にし、ユーザーはAPIコールだけでFMを活用できる。

生成AIのデメリット・課題

ハルシネーション（Hallucination） ★試験頻出

LLMが事実と異なる情報を高い自信を持って生成する現象。最もよく出題される制限事項。

  • 存在しない論文・書籍・人物を「創作」して提示する
  • 計算結果の誤り、日付・固有名詞の誤り
  • 対策①：RAG（Retrieval-Augmented Generation） — 外部知識を参照させる
  • 対策②：根拠情報の引用を明示させるプロンプト設計
  • 対策③：Human-in-the-loop — 人間による出力レビュー
  • 対策④：Bedrock Guardrails — 有害・不正確な出力のフィルタリング
コンテキストウィンドウの制限

一度に処理できるトークン数に上限がある（モデルにより数千〜数百万トークン）。長文ドキュメント全体を一度に処理できない場合がある。

  • 対策①：チャンキング（Chunking） — 文書を適切なサイズに分割して処理
  • 対策②：RAG — 必要な部分のみを検索・抽出して提供
  • 対策③：階層的要約 — 大きなドキュメントを段階的に要約
バイアスと公平性の問題

訓練データに含まれる社会的偏りがモデルの出力に反映される。特定の人種・性別・国籍・宗教への差別的・ステレオタイプ的な出力のリスク。

  • 対策：Bedrock Guardrails でトピックや言語フィルタを設定
  • 対策：モデル評価（Model Evaluation） で定期的にバイアス検査
  • 対策：多様なデータセットでのファインチューニング
著作権・知的財産の問題

訓練データに含まれる著作物の記憶と再現リスク、生成コンテンツの著作権帰属が法的に不明確な部分がある。

プロンプトインジェクション攻撃

悪意あるユーザーがプロンプトに特殊な指示を埋め込み、モデルの本来の動作を迂回させる攻撃。エージェントシステムで特に危険。Bedrock Guardrailsで対策可能。

計算コストの高さ

LLMの推論はGPU/TPU等の専用ハードウェアが必要で計算集約的。特に大規模モデルほどレイテンシとコストが高くなる。適切なモデルサイズの選択が重要。

適切な生成AIモデルの選択基準

選択基準 考慮事項 AWSでの対応
性能・精度 ベンチマークスコア（MMLU: 知識理解、HumanEval: コーディング、TruthfulQA: 事実性） Bedrock Model Evaluation でベンチマーク比較
モダリティ テキスト / 画像 / コード / マルチモーダル（テキスト+画像）のどれが必要か Bedrock の各モデルを用途別に選択
コンテキスト長 1回のリクエストで処理する最大トークン数。長文処理には大コンテキスト対応モデルが必要。 Claude 3: 最大200Kトークン（200,000）
レイテンシ リアルタイム応答（チャット：<1秒）vs バッチ処理（分〜時間単位） 軽量モデル（Claude Haiku）でレイテンシを低減
コスト 入出力トークン単価（$/1Mトークン）。モデルサイズが大きいほど高コスト。 Claude Haiku < Sonnet < Opus の順で高コスト
プライバシー データがモデル訓練に使用されるか、サードパーティと共有されるか Bedrockはデータを訓練に使用せずAWS内で処理
コンプライアンス HIPAA、GDPR、SOC2等の規制要件への対応 AWS各種認定（SOC2、ISO27001、HIPAA等）
カスタマイズ性 ファインチューニング可否、プロンプト学習（Continued Pre-training）対応 Bedrock Fine-tuning で対応

ビジネス価値とメトリクス

ビジネスメトリクス

工数削減率（FTE削減）
生成AIによって削減できる人件費換算。コスト削減効果の定量化。
処理速度向上
スループット（単位時間の処理件数）の増加率。生産性向上の指標。
顧客満足度
CSAT（顧客満足スコア）/ NPS（推奨度スコア）。サービス品質の指標。
品質向上
エラー率・バグ率の低下。コンテンツ品質スコアの向上。
収益貢献
コンバージョン率向上、アップセル率。ROI（投資対効果）の測定。

技術メトリクス

TTFT（Time-to-First-Token）
最初のトークンが返るまでの時間。インタラクティブ体験の重要指標。
スループット（tokens/sec）
1秒あたりの生成トークン数。高いほどユーザー体験が良い。
コスト（$/1M tokens）
入力・出力トークン単価。コスト最適化の基本指標。
BLEU / ROUGE スコア
翻訳・要約の品質評価スコア。参照テキストとの類似度で測定。
BERTScore
BERT埋め込みを使った意味的類似度評価。BLEU/ROUGEより精度が高い。
有害率（Toxicity Rate）
有害・不適切な出力の割合。ガードレールの有効性評価に使用。
試験対策ポイント：モデル選択の考え方

AIF-C01では「どのモデル・サービスを選ぶべきか」という状況問題が多く出ます。コスト重視 → 小さいモデル（Claude Haiku）、精度重視 → 大きいモデル（Claude Opus）、コード生成 → Amazon Q Developer、RAG → Bedrock Knowledge Bases + Titan Embeddingsという基本パターンを覚えておきましょう。

3-3 生成AIアプリケーション構築のためのAWSサービス

Amazon Bedrock（最重要サービス）

Amazon Bedrockとは

Amazon Bedrockは、複数の基盤モデル（FM）への統一APIアクセスを提供するマネージドサービスです。インフラ管理不要で、Anthropic、Meta、Stability AI などの多数のFMをAPIコール1本で利用できます。訓練データはAWSがモデル学習に使用せず、データはAWSインフラ内で処理されます。

Bedrockの主要コンポーネント

🤖Bedrock Agents
  • マルチステップタスクの自動化
  • 外部APIやAWSサービスとの連携
  • ReAct（Reasoning + Acting）フレームワーク
  • カスタムツール（Lambda関数）の呼び出し
  • 自律的なタスク実行エージェント構築
📚Bedrock Knowledge Bases
  • RAGのフルマネージド実装
  • S3のドキュメントを自動インデックス化
  • ベクターデータベース管理（OpenSearch, Aurora等）
  • セマンティック検索による関連チャンク取得
  • 引用付き回答生成のサポート
🛡Bedrock Guardrails
  • 有害コンテンツのフィルタリング
  • 禁止トピックの設定
  • PII（個人情報）の自動マスキング
  • プロンプトインジェクション対策
  • グラウンディング（根拠確認）チェック
📊Bedrock Model Evaluation
  • モデル間のパフォーマンス比較
  • 自動評価（BERTScore, ROUGE等）
  • 人間評価ワークフロー
  • バイアス・有害性のテスト
  • カスタムデータセットでの評価
⚙Bedrock Fine-tuning
  • カスタムデータでのモデル特化
  • Continued Pre-training（継続事前学習）
  • Supervised Fine-tuning（教師あり）
  • S3のトレーニングデータを使用
  • プロビジョンドスループットでデプロイ
🎨Bedrock Studio
  • ノーコードFMプロトタイピング
  • Agents・Knowledge Bases のビジュアル構築
  • チームでのコラボレーション
  • 迅速なPoC（概念実証）開発

AWSを使う生成AIのメリット
  • セキュリティ： データはAWSインフラ内で処理され、モデル訓練に使用されない
  • スケーラビリティ： オンデマンドのキャパシティで需要に応じて自動スケール
  • 統合： S3、Lambda、CloudWatch 等の既存AWSサービスとシームレス連携
  • コンプライアンス： SOC2、ISO27001、HIPAA、GDPR等の各種認定取得済み
  • マルチモデル： 複数FMを同一APIで利用可能。ベンダーロックインを回避
  • マネージド運用： インフラ管理不要でFMの活用に集中できる

コストと最適化

Bedrockの料金体系（オンデマンド）
コスト = (入力トークン数 × 入力単価) + (出力トークン数 × 出力単価)

例：Claude 3 Sonnet の場合 入力：$3.00 / 1M tokens 出力：$15.00 / 1M tokens

※ 出力トークンは入力より高単価になるケースが多い

コスト最適化の戦略

戦略 説明 効果
モデルの選択最適化 タスクの複雑さに応じたモデル選択。単純タスクには軽量モデル（Claude Haiku）を使用。 最大10倍のコスト差
プロンプト最適化 不要な文字・説明を削除してトークン数を削減。Few-shotの例を減らす。 10〜50%のコスト削減
キャッシング活用 頻繁に使用されるプロンプト・回答をElastiCacheやDynamoDBにキャッシュ。 繰り返しリクエストのコストゼロ化
プロビジョンドスループット 一定のスループットを事前予約。予測可能な大規模利用では安定したコストとパフォーマンス。 オンデマンド比で最大50%削減
バッチAPI活用 リアルタイム不要な処理はBatch Inferenceで処理。非ピーク時に実行。 オンデマンド比50%割引
Bedrockの料金体系のポイント：
オンデマンド：使用量に応じた従量課金。予測困難な利用量に最適。
プロビジョンドスループット：一定のスループット（モデルユニット）を1時間/1ヶ月単位で予約。安定した大規模利用に最適でコスト削減効果が高い。
Batch Inference：大量リクエストを一括処理。リアルタイム不要のタスクに最適でコストが最も安い。

練習問題　（全6問）

問題 1

生成AIが「ハルシネーション」を起こすとはどのような状態を指しますか？また、その最も効果的な対策として正しいものを選んでください。

  1. モデルが回答を生成するのに時間がかかりすぎる状態。対策はGPUのスペックアップ。
  2. モデルが事実と異なる情報を高い自信を持って生成する状態。対策としてRAGによる外部知識の参照が有効。
  3. モデルが入力プロンプトを理解できず空の回答を返す状態。対策はプロンプトの短縮。
  4. モデルが有害なコンテンツを生成する状態。対策はモデルの再訓練のみ。
正解：B

ハルシネーションは生成AIの最も重要な制限事項の一つで、モデルが事実と異なる情報（存在しない人物・論文・日付など）を自信を持って出力する現象です。最も効果的な対策はRAG（Retrieval-Augmented Generation）で、外部の信頼できるデータソースから関連情報を検索し根拠として回答させることでハルシネーションを抑制します。AWSでは Bedrock Knowledge Bases がRAGのマネージド実装を提供します。

問題 2

ある企業が社内のPDF文書（数千件）を対象に、従業員がチャットで質問できるシステムを構築したいと考えています。最も適切なAWSアーキテクチャはどれですか？

  1. Amazon Translateで全文書を英語に翻訳してからAmazon Comprehendで検索する。
  2. Amazon EC2にLLMをセルフホストし、全文書をコンテキストウィンドウに詰め込む。
  3. Amazon S3に文書を保存し、Amazon Bedrock Knowledge Basesでインデックス化。Bedrock のFMと組み合わせたRAGアーキテクチャで回答生成する。
  4. Amazon Kendra単体で構築し、LLMを使用しないキーワード検索システムにする。
正解：C

社内文書へのチャット型QAシステムはRAGアーキテクチャが最適解です。S3でPDF保存、Bedrock Knowledge BasesでRAGインデックス化、Bedrock FMで根拠に基づいた回答生成を行います。

B（誤）： コンテキストウィンドウへの全文書詰め込みは数千件には非現実的でコスト・レイテンシの問題があります。

D（誤）： LLMなしでは自然な日本語での回答生成ができません。

問題 3

LLMの「コンテキストウィンドウ」に関する説明として正しいものはどれですか？

  1. モデルが画面に表示できるウィンドウの最大サイズを指す。
  2. モデルの学習に使用したデータセットの総サイズを指す。
  3. モデルが一度のリクエストで処理できる最大トークン数を指し、これを超えると処理できない。
  4. モデルが同時に処理できる並列リクエスト数の上限を指す。
正解：C

コンテキストウィンドウは、LLMが1回のリクエスト（入力＋出力）で扱えるトークンの最大数です。これを超えた場合の対策としてチャンキング・RAG・階層的要約が有効です。また、コンテキストウィンドウのトークン数がコストに直結するため、プロンプト設計でのコスト最適化も重要です。

問題 4

Bedrockの「プロビジョンドスループット」と「オンデマンド」の主な違いとして正しいものはどれですか？

  1. プロビジョンドスループットはFinetuning済みモデルにしか使用できない。オンデマンドはBase Modelのみ。
  2. プロビジョンドスループットは一定のスループットを事前予約するため安定したパフォーマンスとコスト削減が実現でき、オンデマンドは従量課金で使用量が予測しにくい場合に適している。
  3. プロビジョンドスループットはAmazon以外のモデルには使用できない。
  4. オンデマンドはGuardrailsを設定できないが、プロビジョンドスループットは設定できる。
正解：B

オンデマンドは使用量に応じた従量課金でPoC・開発・不定期処理に最適。プロビジョンドスループットは予約したモデルユニット×時間で課金し、大規模・定常処理で最大50%安くなり、スループットも保証されます。Fine-tuningしたカスタムモデルを本番利用する場合はプロビジョンドスループットが必須です。

問題 5

生成AIモデルのアーキテクチャに関して、「拡散モデル（Diffusion Models）」の説明として最も正しいものはどれですか？

  1. 生成器と識別器が競い合うことでリアルなデータを生成するアーキテクチャ。
  2. データを潜在空間に圧縮・再構成し、潜在空間のサンプリングで新データを生成するアーキテクチャ。
  3. 自己注意機構によって文脈全体を並列処理し、次のトークンを予測するアーキテクチャ。
  4. データに徐々にノイズを加える「順拡散」プロセスと、そのノイズを段階的に除去する「逆拡散」プロセスを学習することで高品質な画像を生成するアーキテクチャ。
正解：D

A（誤）： GAN（敵対的生成ネットワーク）の説明。B（誤）： VAE（変分オートエンコーダー）の説明。C（誤）： Transformer（LLM）の説明。

D（正）： 拡散モデルはStable Diffusion・DALL-E 3・Amazon Titan Image Generatorの基盤技術。GANの訓練不安定性（mode collapse）を解消し、より多様で高品質なサンプルを生成できます。

問題 6

あるEコマース企業が生成AIを活用して以下を実現したいと考えています。①開発者のコーディング支援（コード補完・レビュー）、②顧客向けチャットボット（商品QA）、③大量商品の説明文生成（バッチ処理）。最もコスト効率の良いAWSサービスの組み合わせはどれですか？

  1. ① SageMaker JumpStart、② Amazon Kendra単体、③ Amazon Translate
  2. ① Amazon CodeGuru、② Amazon Lex、③ Amazon Comprehend
  3. ① Amazon Q Developer、② Bedrock Knowledge Bases + Claude（Haiku）+ Guardrails、③ Bedrock Batch Inference（Claude Haiku）
  4. ① Bedrock Claude Opus、② Bedrock Claude Opus、③ Bedrock Claude Opus（オンデマンド）
正解：C

① Amazon Q Developer：IDE統合によるコード補完・生成・セキュリティスキャンに特化したサービス。

② Bedrock Knowledge Bases + Claude Haiku + Guardrails：商品QAにはRAGが必要。リアルタイム応答にはコスト低・低レイテンシのHaikuが最適。

③ Bedrock Batch Inference（Haiku）：バッチ処理はオンデマンドより50%安価。シンプルな定型文生成はHaikuで十分。

D（誤）： 全てをClaude Opusのオンデマンドで処理するとコストが最高になります。
