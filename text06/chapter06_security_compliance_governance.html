6-1 AIシステムの保護

AIシステムはデータ収集・モデル訓練・推論という独自のライフサイクルを持ち、各フェーズに特有のセキュリティリスクが存在します。AWS は IAM、KMS、VPC、CloudTrail などの基盤サービスを組み合わせて、エンドツーエンドの保護を実現します。

AIF-C01 では「どのシナリオにどのAWSサービスを使うか」が問われます。各サービスの主要ユースケースと他サービスとの組み合わせパターンを理解してください。

AIシステムを保護するためのAWSのサービスと機能 — 概要

AIシステムのセキュリティは以下の4つの観点で整理できます。

観点 主要AWSサービス 目的
データ保護 Amazon Macie, AWS KMS 機密データの検出・暗号化
アクセス制御 AWS IAM, Lake Formation 最小権限の実施・細粒度認可
ネットワーク保護 Amazon VPC, セキュリティグループ 通信の隔離・暗号化
検知・対応 CloudTrail, GuardDuty, Inspector, Security Hub 脅威検知・監査証跡

Amazon Macie / AWS IAM / AWS KMS

Amazon Macie

機械学習を活用して S3 バケット内の機密データを自動検出・分類するサービスです。AIシステムの ML データセットに含まれる個人情報の特定に不可欠です。

  • PII 検出対象: 氏名、メールアドレス、クレジットカード番号、マイナンバー、パスポート番号、銀行口座番号など
  • 自動的なバケット評価: パブリック公開・暗号化なし・外部共有バケットを自動フラグ
  • ML データセットスキャン: 訓練データに混入した機密情報を検出
  • コンプライアンス監査レポートの自動生成: GDPR・HIPAA 対応に活用
  • EventBridge 連携: PII 検出時に自動修復ワークフローを起動

AWS Identity and Access Management (IAM)

AWSリソースへのアクセスを 最小権限の原則 に基づいて制御します。SageMaker での主要なロール設計は以下の通りです。

IAM ロール用途主な許可ポリシー
SageMaker サービスロール SageMaker が AWS サービスを呼び出す S3 読み書き、ECR イメージ取得
実行ロール (Training) トレーニングジョブの実行コンテナ S3 データアクセス、CloudWatch Logs 書き込み
エンドポイント呼び出しロール 推論 API の呼び出し権限 sagemaker:InvokeEndpoint
Bedrock 実行ロール Bedrock API の呼び出し bedrock:InvokeModel、S3 アクセス (RAG用)

重要な IAM 条件キー:

  • sagemaker:VolumeKmsMasterKey — KMS キーの強制
  • sagemaker:NetworkIsolation — ネットワーク隔離の強制
  • aws:SourceVpc — VPC からのアクセスのみ許可
  • aws:SecureTransport — HTTPS 通信のみ許可

AWS Key Management Service (KMS)

暗号化キーの一元管理により、保存時・転送時のデータ保護を実現します。

  • 保存時暗号化の対象: S3 オブジェクト、SageMaker ノートブックインスタンス、EBS ボリューム、SageMaker Feature Store、SageMaker Model Registry
  • キーポリシー: IAM ポリシーより優先して評価されるリソースベースのポリシー
  • 自動キーローテーション: CMK を年 1 回自動ローテーション (AWS マネージドキーは自動、CMK は有効化が必要)
  • マルチリージョンキー: 同一キーマテリアルを複数リージョンで使用し、DR 対応を容易化
  • CloudTrail 連携: 全キー使用履歴を記録しコンプライアンス証跡として活用
S3 暗号化オプションの比較
SSE-S3 (AES-256): AWSがキー管理、追加料金なし SSE-KMS : KMS CMK を使用、キー使用の CloudTrail 記録あり ← 推奨 SSE-C : クライアントがキーを提供、AWSはキーを保持しない クライアント側暗号化: S3 に到達する前に暗号化、最高レベルの制御
SSE-KMS は CloudTrail にキー使用ログが残るため、監査証跡が必要なコンプライアンス要件に最適です。SSE-S3 より監査性が高い点を試験では選択の根拠にしてください。

Amazon VPC / AWS CloudTrail / Amazon Inspector

Amazon VPC (SageMaker での活用)

SageMaker を VPC 内で実行することで、MLデータがインターネットを経由しない安全な環境を構築できます。

  • VPC 内でのトレーニングジョブ実行: S3 や他の AWS サービスとの通信をプライベートに保つ
  • VPC エンドポイント (PrivateLink):
    • S3 ゲートウェイエンドポイント: S3 へのプライベートアクセス
    • SageMaker API エンドポイント: SageMaker コントロールプレーンへのプライベートアクセス
    • Bedrock インターフェイスエンドポイント: Bedrock API のプライベートアクセス
  • セキュリティグループ: インバウンド/アウトバウンドトラフィックをポート・プロトコル・送信元 IP で制御
  • ネットワーク隔離 (Network Isolation): トレーニングコンテナからの一切の外部通信を遮断 (EnableNetworkIsolation: true)
  • NACLs (ネットワーク ACL): サブネットレベルの追加防御層

AWS CloudTrail

AWS アカウント内の全 API コールを記録するサービスです。AI/ML システムのコンプライアンス監査の根幹を担います。

  • 記録対象: SageMaker、Bedrock、Rekognition、Comprehend、すべての AWS AI サービス
  • 記録内容: 誰が・いつ・どのリソースに・何の操作をしたか + レスポンス
  • S3 への長期保存: 監査証跡として 1〜7 年間の保持を推奨
  • CloudWatch Logs 連携: リアルタイム監視・異常アクティビティへの自動アラート
  • CloudTrail Lake: SQL クエリによるマネージド監査ログ分析 (Athena 不要)
  • Athena との連携: S3 上のログを SQL クエリで分析 (コスト効率が高い)
  • 整合性検証: ログファイルのハッシュ検証により改ざん検知

Amazon Inspector

EC2・Lambda・ECR コンテナの脆弱性を自動スキャンするサービスです。

  • SageMaker コンテナの脆弱性検出: ECR に push されたカスタムコンテナの CVE スキャン
  • 継続的スキャン: 新しい CVE が公開されると自動的に再評価
  • CVSS スコア: 脆弱性の深刻度を定量評価 (0〜10)
  • Security Hub 連携: 検出結果を一元的なダッシュボードで管理

AWS Security Hub / Amazon GuardDuty / Bedrock Guardrails

AWS Security Hub

  • セキュリティアラートの一元管理ダッシュボード
  • AWS 基礎セキュリティのベストプラクティス準拠確認
  • GuardDuty・Inspector・Macie の結果を統合
  • 自動修復ワークフロー (Security Hub Automations)
  • クロスアカウント・クロスリージョン集約

Amazon GuardDuty

  • 脅威インテリジェンスによる機械学習ベースの異常検知
  • S3 バケットへの不正アクセス検知
  • 異常な IAM アクティビティの検知
  • GuardDuty for ML: SageMaker の異常 API コール検知
  • EventBridge 連携による自動インシデント対応

Amazon Bedrock Guardrails (セキュリティ観点)

生成 AI アプリケーションのセキュリティとコンプライアンスをコード変更なしで実施できるガードレール機能です。

  • プロンプトインジェクション防御: 悪意あるユーザーがシステムプロンプトを上書きしようとする攻撃を検知・ブロック
  • 機密情報の PII フィルタリング: 入力・出力両方向で氏名、SSN、クレジットカード番号等を自動マスキング
  • カスタムデニーリスト: 禁止ワード・トピックをカスタム設定でブロック (競合他社名、機密プロジェクト名等)
  • グラウンディング検証: 応答が参照ドキュメントに基づいているかの確認 (ハルシネーション抑制)
  • コンテンツフィルタリング: 有害・不適切なコンテンツの 6 カテゴリフィルタリング
Bedrock Guardrails はモデル自体の変更ではなく、入出力層に適用されるレイヤーです。すべての Bedrock 対応モデルに適用でき、Fine-tuning されたモデルにも有効です。

ソース引用とデータ出典の文書化

データリネージとトレーサビリティ

AI システムが信頼できる出力を生成していることを証明するためには、データの出所から最終的な推論結果までの全経路を追跡できる仕組みが必要です。

  • Bedrock Knowledge Bases: RAG 応答に引用元チャンクを自動付与。どのドキュメントのどのページが根拠かをトレース可能
  • モデルカード (Model Cards — SageMaker): モデルの訓練データ・意図・評価結果・制限事項・倫理的考慮事項を標準フォーマットで文書化
  • AWS Glue Data Catalog: データセットのスキーマ・メタデータ・データオーナー・更新履歴を中央管理
  • Amazon SageMaker Feature Store: 特徴量の作成者・時刻・リネージ (どのデータソースから生成されたか) を追跡
  • SageMaker Experiments: 実験の設定・データ・コード・メトリクスをリネージグラフとして可視化
著作権管理の観点: 生成 AI の出力が著作権で保護されたソースを使用しているかを記録することは、企業の法的リスク管理において重要です。Bedrock の引用付き応答は、この課題への一つの対応手段です。

安全なデータエンジニアリングのベストプラクティス

1. データの暗号化

転送時暗号化の強制 (S3 バケットポリシー)
{ "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": "arn:aws:s3:::my-ml-bucket/*", "Condition": { "Bool": { "aws:SecureTransport": "false" } } } — TLS 1.2 以上を強制。HTTP アクセスを拒否

2. データアクセス制御

制御手段粒度ユースケース
S3 バケットポリシー バケット・オブジェクト パブリックアクセスブロック、クロスアカウント制御
VPC エンドポイントポリシー サービス・リソース 特定 S3 バケットへのアクセスのみ許可
IAM リソースポリシー API アクション 最小権限、MFA 条件付きアクセス
AWS Lake Formation 列・行レベル 分析チームごとに参照可能な列を制限

3. データの匿名化・仮名化

Amazon Macie

  • S3 内 PII の自動検出
  • 機密レベルの分類・レポート
  • EventBridge で自動修復トリガー

AWS Glue + Comprehend

  • Glue ETL でデータマスキング変換
  • Comprehend で PII エンティティ検出
  • Comprehend の自動マスキング機能

4. データ保持ポリシー

  • S3 Lifecycle Policy: 一定期間後の自動削除・低コストストレージ層への移行
  • S3 Object Lock (WORM): コンプライアンス要件による削除防止・改ざん防止
  • AWS Backup: バックアップポリシーの一元管理・クロスリージョンバックアップ
  • Amazon Macie + EventBridge: 保持期限超過データの自動通知・削除トリガー

AIシステムのセキュリティとプライバシーに関する考慮事項

モデルに対するセキュリティ攻撃の分類

攻撃の種類内容対策手段
データポイズニング 悪意あるデータを訓練セットに注入し、モデルの挙動を意図的に操作する 訓練データの出所検証、Macie によるデータ品質スキャン
モデル抽出攻撃 大量の API クエリを通じてモデルの機能を模倣したコピーモデルを作成する API レート制限、GuardDuty による異常呼び出し検知
敵対的サンプル (Adversarial Examples) 人間には識別困難な微細な変更を加えた入力で意図的に誤分類させる 入力検証・サニタイゼーション、モデルの堅牢性向上
プロンプトインジェクション LLM のシステムプロンプトや指示を上書きする悪意ある入力を注入する Bedrock Guardrails、入力検証、最小権限のシステムプロンプト設計
モデルインバージョン モデルの出力から訓練データ (特に PII) を逆算・推測する 差分プライバシー、訓練データの匿名化
メンバーシップ推測 特定のデータが訓練セットに含まれていたかどうかを推測する 差分プライバシー、訓練データアクセスの厳格な管理

プライバシー保護技術

差分プライバシー (Differential Privacy)

  • 出力にノイズを追加し個人情報の推測を困難にする
  • 数学的なプライバシー保証 (ε の設定)
  • 集計統計・モデル訓練に適用

連合学習 (Federated Learning)

  • データを中央サーバーに送らずモデルを分散学習
  • 各デバイス/組織でローカルに訓練しモデル重みのみを共有
  • 医療・金融分野での活用事例

AWS 固有のプライバシー保護

Amazon Bedrock のデータプライバシー: デフォルトでは、ユーザーの入力データはモデルの再学習・改善に使用されません。データは AWS リージョン内で処理され、Bedrock の推論処理は AWS インフラ内に留まります。
  • AWS Artifact: AWS のコンプライアンスレポートをセルフサービスでダウンロード。監査人への提出資料として活用
  • AWS データ常駐: リージョン選択によりデータが特定地理的区域内に留まることを保証
  • 同型暗号 (研究段階): 暗号化したまま計算を行い、クラウドプロバイダにも平文データを見せない技術

6-2 AIシステムのガバナンスとコンプライアンス規制

AIシステムの普及に伴い、各国・業界で規制フレームワークが整備されています。AWS はこれらの規制要件を満たすための認証取得済みサービスと、コンプライアンスを自動化・可視化するサービス群を提供しています。

GDPR 第 22 条 — 自動化された意思決定と AI

AIシステムが個人に対して法的効力または重大な影響を与える自動的意思決定を行う場合、その個人は人間による審査を要求する権利があります。試験では、「GDPR に準拠した AIシステムが持つべき機能」として頻出です。

  • ローン審査 AI による否決 → 人間審査の要求権
  • 採用 AI による書類選考落ち → 理由の説明と人間審査の要求権
  • 対応手段: SageMaker Clarify によるモデル説明可能性、人間参加型ワークフロー (Amazon Augmented AI / A2I)

HIPAA — PHI (Protected Health Information) の 18 識別子

HIPAA が保護対象とする PHI には以下の 18 の識別子が含まれます。Comprehend Medical はこれらを電子カルテ等のテキストから自動検出します。

PHI 18 識別子 (抜粋)
氏名 / 地理的データ / 日付 (誕生日・入院日等) / 電話番号 / FAX番号 メールアドレス / SSN / カルテ番号 / 健康保険番号 / 口座番号 証明書番号 / 自動車登録番号 / デバイス識別子 / URL / IPアドレス 指紋・声紋等の生体情報 / 顔写真 / その他固有の識別子

ガバナンスと規制コンプライアンスを支援するAWSのサービスと機能

AWS Config

AWS リソースの設定を継続的に記録・評価し、コンプライアンス状態をリアルタイムで可視化します。

  • コンフォーマンスパック (Conformance Packs): HIPAA・PCI DSS・GDPR・CIS Benchmark 等の業界フレームワークに対応したルールセットをワンクリックで展開
  • マネージドルール例:
    • s3-bucket-server-side-encryption-enabled: S3 暗号化を確認
    • sagemaker-notebook-no-direct-internet-access: ノートブックの直接インターネットアクセスを検出
    • sagemaker-endpoint-config-kms-key-configured: エンドポイントの KMS 暗号化を確認
  • Configuration History: 誰が・いつ・どのように設定を変更したかを時系列で追跡
  • 自動修復 (Auto Remediation): 非準拠リソースを SSM Automation で自動修復

AWS Audit Manager

監査の準備作業を自動化し、コンプライアンス評価のエビデンスを継続的に収集します。

  • 対応フレームワーク: GDPR、HIPAA、PCI DSS、SOC 2、ISO 27001、NIST CSF、FedRAMP 等
  • エビデンスの自動収集: AWS Config、CloudTrail、Security Hub から自動的にエビデンスを収集・整理
  • カスタムフレームワーク: 自社ポリシーに基づくカスタム評価フレームワークの作成
  • 監査レポートの自動生成: 外部監査人への提出用レポートをワンクリック生成
  • 評価の継続性: 四半期ごとの監査ではなく、常時エビデンスを蓄積

AWS Artifact

AWS 自身のコンプライアンスレポートとアグリーメントにセルフサービスでアクセスできるポータルです。

  • 入手可能なドキュメント: ISO 認定書 (ISO 27001/27017/27018)、SOC 1/2/3 レポート、PCI DSS AOC、HIPAA ホワイトペーパー、FedRAMP 認定証 等
  • NDA 管理: 機密レポートへのアクセスに必要な NDA を電子署名で締結
  • AWS Artifact Agreements: BAA (HIPAA)、GDPR DPA 等のアグリーメントをコンソールで締結・管理

SageMaker Model Cards

  • モデルの透明性文書化
  • 記載: 目的、訓練データ、評価結果、制限事項、倫理的考慮
  • Model Registry と統合
  • EU AI 法の技術文書要件への対応

SageMaker Governance (Model Dashboard)

  • 全モデルの一元的な可視化
  • リスクレーティングの付与
  • コンプライアンスステータスの追跡
  • ドリフト・バイアスアラートの集約

AWS CloudTrail + Amazon CloudWatch の組み合わせ

すべての AWS AI/ML API アクティビティを記録・監視する基盤です。

  • CloudTrail → CloudWatch Logs: API ログをリアルタイムで CloudWatch に転送
  • CloudWatch Metric Filter: 特定パターン (例: Bedrock モデル呼び出しの急増、Root 利用者ログイン) を検出してアラーム作成
  • CloudWatch Logs Insights: ログを SQL ライクなクエリで分析
  • CloudWatch Contributor Insights: 最も多くリソースを消費しているユーザー/IP を自動特定
  • インシデント対応証跡: セキュリティインシデント発生時の完全な操作履歴を提供
試験では「コンプライアンス証明のために監査人にどのドキュメントを提供するか」という問いに対し AWS Artifact、「自社リソースの設定コンプライアンスを継続的に評価する」には AWS Config、「監査エビデンスを自動収集する」には AWS Audit Manager を選択してください。

データガバナンスフレームワークの構成要素

構成要素AWSサービス機能
データカタログ AWS Glue Data Catalog データセットのメタデータ (スキーマ・所有者・更新日・分類) を中央管理
データ品質 AWS Glue Data Quality データ品質ルールの自動チェック・スコアリング (完全性・正確性・一貫性)

AWS Lake Formation — データレイクのセキュリティ一元管理

AWS Lake Formation は Glue Data Catalog と統合し、データレイク全体の細粒度アクセス制御を提供します。

  • テーブルレベルアクセス制御: 特定のデータベース・テーブルへのアクセスをロール単位で付与
  • 列レベルセキュリティ (Column-Level Security): 機密列 (SSN、給与等) を特定ロールのみに公開
  • 行レベルフィルタリング (Row Filter): 部門・地域・ユーザー属性に応じたデータフィルタリング
  • データセルフィルター: 列 + 行フィルターの組み合わせで精密なアクセス制御
  • ガバナンスレポート: 誰がどのデータにアクセスしたかの監査ログ
  • クロスアカウント共有: RAM (Resource Access Manager) を使った組織横断のデータ共有

データメッシュアーキテクチャ

大規模組織における分散型データガバナンスアーキテクチャです。

  • ドメイン主導のデータ所有: 各事業部門が自身のデータに対するオーナーシップと責任を持つ
  • データプロダクトとしての提供: 各ドメインが発見可能・信頼できる・自己記述的なデータプロダクトを公開
  • 連合したガバナンス: 中央チームが全体ポリシーを定め、各ドメームが自律的に実施
  • AWS での実装: Lake Formation (中央ガバナンス) + Glue Data Catalog (データカタログ) + RAM (クロスアカウント共有)

ガバナンスプロトコルに従うためのプロセス — モデルガバナンスのライフサイクル

モデル開発標準の策定

  • 訓練データの出所確認・ライセンス確認プロセス
  • セキュリティレビューのチェックリスト策定

モデルレジストリによる一元管理

  • SageMaker Model Registry: バージョン管理・承認フロー・メタデータ保持
  • 記録項目: 精度・バイアスメトリクス・訓練データ情報・Model Cards リンク

  • コンテナイメージのバージョン管理と再現性の確保

  • 本番デプロイ前のゲートレビュー (技術・法務・倫理の各観点)
  • ステークホルダー承認プロセス (Product Owner・Security・Compliance)
  • Model Cards の提出と審査
  • SageMaker Pipelines の RegisterModel ステップとの統合

継続的モニタリング

  • SageMaker Model Monitor: データドリフト・予測品質・バイアスドリフト・説明可能性ドリフトを定期監視
  • 閾値設定: バイアスメトリクスが閾値を超えた場合に自動アラート
  • 自動再学習トリガー: ドリフト検知 → EventBridge → SageMaker Pipeline 起動
  • SageMaker Clarify: バイアスレポートの定期生成

インシデント対応

  • モデルの緊急停止プロセス (エンドポイント削除・トラフィックゼロへの切替)
  • ロールバック手順 (Model Registry の以前バージョンへの切替)
  • 影響を受けたユーザーへの通知 (Amazon SNS + SES)
  • 根本原因分析 (RCA) の実施と再発防止策の策定

定期的なレビューと廃止管理

  • 四半期ごとのモデル性能レビューとコンプライアンス監査
  • バイアス評価の定期的再実施 (データ分布の変化に対応)
  • モデル廃止プロセス: エンドポイント削除 → アーティファクトのアーカイブ → ドキュメント最終化
  • 後継モデルへの移行計画と依存サービスへの通知

練習問題

各設問を読み、最適な選択肢を選んでから解答を確認してください。

1 ある医療系スタートアップが Amazon SageMaker を使って患者の画像診断 AI を開発しています。訓練データセットには HIPAA が定める PHI が含まれる可能性があります。訓練データが保存されている S3 バケットを自動的にスキャンして PHI を検出し、コンプライアンスレポートを生成するには、どのサービスが最も適切ですか？
  • AWS Config の s3-bucket-server-side-encryption-enabled ルール
  • Amazon Macie
  • Amazon Inspector
  • AWS Security Hub
正解: B — Amazon Macie

Amazon Macie は機械学習を使って S3 バケット内の機密データ (PHI を含む PII) を自動検出・分類し、コンプライアンスレポートを生成するサービスです。

  • A (AWS Config): S3 バケットの設定コンプライアンス (暗号化有無等) を評価しますが、データの内容は検査しません。
  • C (Amazon Inspector): EC2・Lambda・ECR コンテナの CVE 脆弱性をスキャンするサービスで、データ内容の分析は対象外です。
  • D (AWS Security Hub): 各サービスのセキュリティアラートを集約するダッシュボードです。自身は PII を検出しません。
2 大手金融機関が社内の生成 AI チャットボット (Amazon Bedrock ベース) を構築しています。ユーザーが意図的にシステムプロンプトを上書きして機密情報を引き出そうとする攻撃 (プロンプトインジェクション) と、応答に顧客の口座番号が含まれてしまうリスクの両方を防ぐには、どのサービスの機能が最も直接的に対応できますか？
  • Amazon GuardDuty for Machine Learning
  • AWS WAF のマネージドルールグループ
  • Amazon Bedrock Guardrails
  • SageMaker Model Monitor のデータ品質モニタリング
正解: C — Amazon Bedrock Guardrails

Amazon Bedrock Guardrails はプロンプトインジェクション防御と PII フィルタリング (入出力両方向) を単一の機能として提供します。コード変更なしで全 Bedrock 対応モデルに適用できます。

  • A (GuardDuty for ML): SageMaker の異常 API 呼び出しを検知しますが、プロンプト内容のフィルタリングは行いません。
  • B (AWS WAF): HTTP レイヤーの Web アプリケーションファイアウォールで、LLM プロンプトのセマンティックな評価はできません。
  • D (Model Monitor): 本番モデルのデータドリフト・品質を監視するサービスです。リアルタイムの入出力フィルタリングは対象外です。
3 EU に本社を置く EC 事業者が機械学習を使ったパーソナライズレコメンデーションエンジンを運用しています。GDPR 第 22 条への準拠のため、ユーザーが自動推薦の決定に異議を申し立てた場合に人間によるレビューを提供できる仕組みが必要です。このユースケースに最も適した AWS サービスはどれですか？
  • AWS Audit Manager の GDPR フレームワーク
  • Amazon SageMaker Clarify のバイアスレポート
  • Amazon Augmented AI (A2I)
  • AWS Config の GDPR コンフォーマンスパック
正解: C — Amazon Augmented AI (A2I)

Amazon Augmented AI (A2I) は ML モデルの予測に対して人間によるレビューワークフローを組み込むサービスです。GDPR 第 22 条が要求する「自動的意思決定に対する人間による審査」の仕組みを実装するのに直接対応します。

  • A (Audit Manager): コンプライアンスエビデンスの収集・整理に特化し、エンドユーザーへの審査フローは提供しません。
  • B (SageMaker Clarify): モデルの説明可能性・バイアス分析レポートを生成しますが、エンドユーザー向けの審査フローではありません。
  • D (AWS Config): リソース設定のコンプライアンスを評価しますが、ユーザー向けの異議申し立てフローとは無関係です。
4 セキュリティ担当者が「SageMaker トレーニングジョブ実行中に、訓練コンテナが外部インターネット上の第三者サービスに接続しようとした」というインシデントを調査しています。このような外部通信を技術的に防ぐための SageMaker の設定はどれですか？
  • SageMaker ノートブックインスタンスのルートアクセスを無効化する
  • KMS カスタムキーで EBS ボリュームを暗号化する
  • CloudTrail を有効化してすべての API コールを S3 に記録する
  • Network Isolation (EnableNetworkIsolation: true) をトレーニングジョブに設定する
正解: D — Network Isolation (EnableNetworkIsolation: true)

ネットワーク隔離 (Network Isolation) を有効にすると、SageMaker トレーニングコンテナからのすべての外部ネットワーク通信が遮断されます。コンテナは S3 (指定のインプット/アウトプット) にのみアクセスでき、インターネットへの通信はブロックされます。

  • A (ルートアクセス無効化): ノートブックインスタンス内でのルート権限の制限であり、トレーニングコンテナのネットワーク制御ではありません。
  • B (KMS 暗号化): データの機密性保護であり、ネットワーク通信制御ではありません。
  • C (CloudTrail): 事後の調査・記録には有効ですが、通信を「防ぐ」制御ではありません。
5 ある企業の CISO が SOC 2 Type II 認定に向けた外部監査に備えています。AWS インフラ層の SOC 2 レポートを取得し、AWS がセキュリティ・可用性・機密性の要件を満たしていることを監査人に証明する最も効率的な方法はどれですか？
  • AWS Config の Conformance Pack で SOC 2 フレームワークを展開して評価レポートを出力する
  • AWS Artifact にアクセスして SOC 2 レポートをダウンロードし監査人に提供する
  • AWS Audit Manager で SOC 2 フレームワークを選択してエビデンスを収集する
  • Amazon Inspector の評価レポートを SOC 2 証拠として提出する
正解: B — AWS Artifact

AWS Artifact は AWS 自身の第三者機関による SOC 2 レポート・ISO 認定書・PCI DSS AOC 等の公式コンプライアンスドキュメントをセルフサービスでダウンロードできるポータルです。

  • A (AWS Config): 自社 AWS リソースの設定コンプライアンスを評価します。AWS インフラ層の SOC 2 証明書は取得できません。
  • C (AWS Audit Manager): 自社アカウントのリソースに対するコンプライアンスエビデンスを収集します。AWS 自身の SOC 2 レポートとは別物です。
  • D (Amazon Inspector): 脆弱性スキャン結果であり、SOC 2 レポートの代替にはなりません。

補足: A と C は自社のリソース・プロセスが SOC 2 要件を満たしていることを証明するためのサービスです。B は AWS インフラ層の SOC 2 準拠を証明します。両者を組み合わせることで完全な SOC 2 対応が可能です。

6 データサイエンティストチームが複数の部門にまたがる分析データレイクを構築しています。マーケティング部門は顧客の購買履歴テーブルにアクセスできますが、給与列や年齢列は参照できないようにする必要があります。また、営業部門は自分の担当リージョンの行データのみ参照できる制御が必要です。このような細粒度アクセス制御を実現するために最適なサービスはどれですか？
  • S3 バケットポリシーで各部門のプレフィックスへのアクセスを制限する
  • IAM ポリシーで各部門のロールに対して Glue テーブルへの許可を設定する
  • AWS Lake Formation で列レベルセキュリティと行フィルターを設定する
  • SageMaker Feature Store のオフラインストアのアクセスポリシーを利用する
正解: C — AWS Lake Formation の列レベルセキュリティと行フィルター

AWS Lake Formation はデータレイクに対して列レベルセキュリティ (Column-Level Security)と行レベルフィルタリング (Row Filter)を提供します。Glue Data Catalog と統合し、テーブル・列・行の組み合わせで精密なアクセス制御が可能です。

  • A (S3 バケットポリシー): S3 オブジェクト (ファイル) レベルの制御であり、列・行レベルの制御はできません。
  • B (IAM ポリシー): テーブルへのアクセス許可は設定できますが、列単位・行単位のフィルタリングは IAM では実現できません。
  • D (SageMaker Feature Store): 特徴量管理に特化したサービスであり、汎用のデータレイクアクセス制御には適していません。
