1. ビジネス価値とAI/MLの関係

学習時間の目安: 25分
このセクションの目標: AI/MLがビジネスにどのような価値をもたらすか、ROIの考え方、そして試験頻出の「どのユースケースにどのAWSサービスを使うか」の判断軸を習得する。

1.1 AI/MLがビジネスに与える5つの価値

コスト削減

繰り返し作業の自動化、人的ミスの削減、設備の予知保全によりコストを下げる。

例: 請求書処理の自動化でコスト80%削減

収益拡大

パーソナライズ推薦、クロスセル・アップセル、価格最適化で売上を増やす。

例: Netflixのレコメンドで解約率35%低下

リスク低減

詐欺検知、品質検査自動化、コンプライアンス監視で損失リスクを減らす。

例: クレジット不正検知で年間数十億円の損失回避

意思決定の高速化

リアルタイム分析・予測により、人間では処理しきれない速度と規模で意思決定を支援する。

例: アルゴリズム取引で1ミリ秒以下の意思決定

新規サービス創出

AIなしでは不可能だったサービスや体験を生み出す。

例: 対話AI、自動運転、医療画像診断支援

1.2 AI/MLプロジェクトのROI（投資対効果）

ROI = (ビジネス価値 - 総コスト) / 総コスト x 100%

ビジネス価値 = コスト削減額 + 収益増加額 + リスク回避額
総コスト = データ収集費 + 開発費 + インフラ費 + 運用費 + 変更管理費
ROI計算例: 製造業の品質検査AI導入

ビジネス価値（年間）:
・不良品流出削減: 5,000万円 / 検査員削減（12人→3人）: 7,200万円
・ライン停止時間削減: 1,800万円 → 合計: 14,000万円

総コスト（年間）:
・初期開発費（3年償却）: 2,000万円 / AWSインフラ費: 600万円
・保守・再訓練費: 400万円 → 合計: 3,000万円

ROI = (14,000 - 3,000) / 3,000 x 100% = 367%

1.4 AI/ML成熟度モデル

企業のAI/ML成熟度（5段階）:

レベル1: 探索期
  PoC（概念実証）段階。少数のパイロット実験。
  AWS: SageMaker Studio Lab（無料）、Jupyter Notebook

レベル2: 導入期
  いくつかのユースケースが本番稼働。手動プロセスが多い。
  AWS: SageMaker エンドポイント、AI Services（Rekognition等）

レベル3: 拡張期
  複数のMLシステムが稼働。再現可能なパイプラインを整備。
  AWS: SageMaker Pipelines、Feature Store、Model Registry

レベル4: 最適化期
  自動化・継続的学習が実現。Model Monitor で自動再訓練トリガー。
  AWS: SageMaker Autopilot、Model Monitor + EventBridge

レベル5: AI-First
  AIが意思決定の中核。組織全体にAI文化が浸透。
  AWS: 全AWSサービスの統合活用 + Amazon Bedrock

2. 小売・EC・マーケティングのユースケース

学習時間の目安: 30分
このセクションの目標: 小売・ECが最もAI活用が進んだ業界の一つであることを理解し、レコメンド・需要予測・価格最適化・顧客離反予測の具体的な実装を学ぶ。

2.1 パーソナライズ推薦システム

ECサイトの「あなたへのおすすめ」「よく一緒に購入されている商品」は推薦システムによって実現されています。Amazonの売上の35%以上が推薦エンジン由来といわれています。

協調フィルタリング

ユーザーベース推薦

「あなたと似た購買履歴のユーザーが買った商品」を推薦。
例: Spotifyの「Discovery Weekly」

コンテンツベース

商品属性ベース推薦

閲覧・購入した商品の属性（ブランド・カテゴリ）が似た商品を推薦。
例: Netflixの同じ監督の映画推薦

リアルタイム

セッションベース推薦

現在のセッション内の行動（直近クリック）から即座に推薦を更新。
例: 閲覧商品に連動した広告

生成AI

対話型ショッピング

「家族4人の夏のバーベキュー用品を教えて」という自然言語クエリへ対応。
AWS: Amazon Bedrock + Personalize

Amazon Personalize の仕組み: S3に購買履歴・閲覧履歴・商品カタログをアップロードするだけで、Amazonが自社ECで使う協調フィルタリングと同じアルゴリズムをマネージドサービスとして利用できます。カスタムMLの知識不要。

2.3 動的価格設定（ダイナミックプライシング）

ダイナミックプライシングの仕組み:

インプット要素:
  ・競合他社の価格（スクレイピング or API）
  ・需給バランス（在庫数・閲覧数）
  ・時間帯・曜日・季節

  ・過去の価格弾力性データ

     ↓  価格最適化モデル（強化学習 or 回帰）

アウトプット:
  各SKUの最適価格を1時間ごとに自動更新

実例:
  ・航空会社: 残席数・予約時期で同じ座席が1万円〜30万円
  ・Uber: サージプライシング（需要急増時に自動値上げ）
  ・Amazon.com: 1日2,500万回の価格変更を実施

3. 金融・保険・リスク管理のユースケース

学習時間の目安: 25分
このセクションの目標: 金融業界における不正検知・信用スコアリング・アルゴリズム取引・保険引受のAI活用と、規制・コンプライアンス上の考慮点を理解する。

3.1 不正検知（Fraud Detection）

クレジットカード詐欺は全世界で年間数兆円規模の損害をもたらします。AIは毎秒数万件のトランザクションをリアルタイムで監視し、0.1秒以内に詐欺の可能性を判定します。

リアルタイム不正検知のアーキテクチャ（AWS例）:

クレジットカード取引
        |
        v
Amazon Kinesis Data Streams（リアルタイムストリーム）
        |
        v
AWS Lambda（前処理・特徴量計算）
  ・過去24時間の利用パターン
  ・地理的な移動速度（不可能な速度での移動）
  ・利用店舗カテゴリの変化
  ・金額の統計的異常
        |
        v
SageMaker リアルタイムエンドポイント（詐欺スコア算出）
        |
        v
スコア > 閾値?
  YES --> トランザクション拒否 + SMS通知
  NO  --> 承認
不正検知のトレードオフ: 検知感度を上げると（閾値を下げると）正常取引を拒否する「誤検知（FP）」が増え、顧客満足度が低下します。逆に感度を下げると詐欺を見逃す「見逃し（FN）」が増えます。ビジネスコストを考慮した閾値設定が重要です。

3.2 信用スコアリングと与信審査

項目 従来の手法 AI活用後
審査時間 数日〜数週間 数秒（自動審査）〜数分
使用データ 年収・勤続年数・担保（数十項目） キャッシュフロー・SNS行動・購買パターン（数百〜数千項目）
承認精度 デフォルト率 3〜5% デフォルト率 1〜2%（大幅改善）
金融疎外問題 信用履歴なし → 与信不可 代替データで審査可能（スマホ決済履歴など）
SageMaker Clarify の重要性: 信用スコアリングモデルは、性別・人種・年齢などの保護属性に基づく差別的判断を行ってはなりません。SageMaker Clarify のバイアス検出機能で定期的に公平性を監視することが法的にも求められます（米国 Equal Credit Opportunity Act 等）。

3.3 保険業界のAI活用

引受審査

リスク評価の自動化

申込フォーム + 公開データ + テレマティクスデータを統合し、保険料を自動算出。自動車保険の運転スコアリング（運転挙動データで保険料変動）。

損害査定

事故画像の自動査定

スマートフォンで撮影した事故写真をAIが解析し修理費を自動見積もり。Amazon Rekognition + SageMaker カスタムモデル。

不正請求検知

保険詐欺の防止

請求パターンの異常検知（同一病院・同一手術の連続請求など）。グラフニューラルネットワークで詐欺グループの関係性を検出。

予知保全

損害予防サービス

IoTセンサーで設備の劣化を予測し故障前に修理を促す。損害発生を未然に防ぐことで損害率を改善。

4. 医療・ヘルスケアのユースケース

学習時間の目安: 25分
このセクションの目標: 医療AIは高い精度と説明可能性が求められる特殊な領域。画像診断支援・創薬・患者管理のユースケースとAWS Healthcareサービスを理解する。

4.1 医療画像診断支援

放射線科医・病理医・眼科医など、「画像を見て診断する」専門医をAIが支援します。AIは疲労を知らず、24時間365日一定の精度で判定できるという強みがあります。

疾患・用途 画像種別 AIの役割 精度例
胸部X線 肺がん検出 X線画像 結節の自動検出・サイズ測定 専門医と同等（AUC 0.97）
眼底写真 糖尿病網膜症 眼底画像 重症度分類（4段階） 眼科医を上回る精度
皮膚がん判定 皮膚写真 良性/悪性の二値分類 皮膚科専門医と同等
病理スライド 癌細胞検出 デジタル病理画像 癌細胞の領域セグメンテーション 病理医の見逃し率50%削減
Amazon HealthLake: FHIR（医療情報標準）に準拠した医療データのストア・変換・分析サービス。電子カルテ(EHR)・検査データ・画像データを統合し、SageMakerと組み合わせてAIモデルを訓練できます。

4.2 臨床意思決定支援（CDSS）

Clinical Decision Support System の役割

医師・看護師が診断・治療方針を決定する際、電子カルテのデータをリアルタイムで分析してアラートや推薦を提供します。

  • 薬剤相互作用チェック: 処方時に危険な薬の組み合わせを自動警告
  • 敗血症早期警告: バイタルサインの変化パターンから敗血症を数時間前に予測
  • 再入院リスク予測: 退院時に再入院リスクが高い患者を特定しフォローアップ計画を立案
  • 診断コーディング支援: 診療記録からICD-10コードを自動提案（Amazon Comprehend Medical）
医療AIの特別な要件:
  • 説明可能性（XAI）: 「なぜがんと診断したか」を医師が理解できる形で提示する必要がある
  • 規制対応: FDA（米国）や厚労省（日本）の認可が必要な場合がある
  • PHI保護: 個人健康情報（Protected Health Information）の厳格な保護（HIPAA準拠）
  • 偽陰性のリスク: 見逃しが患者の命に直結するため、Recall（再現率）を最大化する設計が必須

5.1 製品品質検査の自動化

コンピュータビジョンによる外観検査:

  | ライン流れる           AIモデル（CNN）
  |                        |

AWS構成 (オプション1): IoT カメラ → AWS IoT Greengrass → SageMaker Neo

5.3 物流・配送最適化と倉庫自動化

ルート最適化

配送効率の最大化

100台のトラックが1,000件の荷物を配送する最適ルートを計算。渋滞・時間指定・積載量を考慮。効果: 走行距離15%削減、CO2排出削減。

需要予測

倉庫在庫の事前配置

需要予測に基づき、売れる商品を事前に近くの物流センターに移動。配送時間を大幅短縮。AWS: Amazon Forecast

倉庫自動化

6.2 RAG（検索拡張生成）のビジネス実装

RAG（Retrieval-Augmented Generation）は、LLMの「知識の古さ」と「ハルシネーション（誤情報生成）」を解決する最重要アーキテクチャです。

RAGのアーキテクチャ（Amazon Bedrock + Knowledge Bases）:

社内ドキュメント（PDF・Word・Webページ）
        |
        v
Amazon Bedrock Knowledge Bases（自動インデックス化）
  1. テキストをチャンク（段落）に分割
  2. Titan Embeddings で各チャンクをベクトル化
  3. Amazon OpenSearch / Aurora pgvector に保存

        ユーザーの質問
              |
              v
        質問をベクトル化
              |
              v
 類似チャンク検索（ベクトル類似度）
              |
              v
  「質問 + 関連チャンク（コンテキスト）」を Claude/Titan へ送信
              |
              v
    根拠に基づいた正確な回答 + 出典URL

利点: 社内最新情報に基づく回答 / ハルシネーション大幅削減 / 出典明記
実装例: 社内ナレッジベースQ&Aシステム
課題: 従業員が社内規定・手続きマニュアルを検索するのに平均20分かかっている。HR部門への同じ質問が月500件。
解決策: Bedrock Knowledge Bases に社内規定PDFを登録し、Slackボットとして公開
効果: 検索時間が20分 → 10秒に短縮、HR担当者の問い合わせ対応業務を70%削減

6.3 Amazon Bedrock エージェント

Bedrock エージェントとは

LLMが自律的に考え → ツールを呼び出し → 結果を評価 → 再度考えるというループで複雑なタスクを実行します。これをReAct（Reasoning + Acting）パターンといいます。

Bedrock エージェントの動作例（旅行予約エージェント）:

ユーザー:「来週の東京出張を手配して。
          予算は8万円以内で航空券と宿を取って。」

エージェントの思考ループ:
  思考1: 航空券の空き状況を確認する必要がある
  行動1: [航空券API] を呼び出す → 候補3件取得
  観察1: JAL 12,000円、ANA 14,000円、LCC 7,800円 が存在

  思考2: 予算内で最適な組み合わせを探す
  行動2: [ホテル検索API] を呼び出す → 新宿周辺3件取得
  観察2: 2泊で 35,000円〜55,000円 の選択肢

  思考3: LCC + ビジネスホテル = 7,800 + 40,000 = 47,800円
         予算内に収まる。ユーザーに確認する。

  最終回答:「LCCで往復7,800円、ホテルAに2泊40,000円の
            合計47,800円のプランを提案します。ご確認ください。」

6.5 生成AIのリスクと責任ある利用

ビジネス展開時の主要リスク:
  • ハルシネーション: LLMが事実でない情報を自信を持って生成する。RAGと出典明記で対策。
  • 著作権侵害: 生成コンテンツが訓練データを無断複製する可能性。Bedrock の Titan モデルはIndemnity（補償）を提供。
  • プロンプトインジェクション: 悪意あるユーザーがプロンプトを操作してシステムを悪用。入力バリデーションとGuardrailsで対策。
  • バイアスと公平性: 訓練データのバイアスが出力に反映される。多様なデータと定期的な評価で対策。
  • Amazon Bedrock Guardrails: トピックフィルター・センシティブ情報マスキング・ハルシネーション検出を設定できる安全装置。

7.2 ユースケース選択の思考フロー

「このビジネス課題に何を使うか」判断フロー:

  [1] 画像・動画を扱う?
          YES --> Rekognition (汎用) / Lookout for Vision (製造品質)
  [2] テキスト・言語を扱う?
          YES --> Comprehend (分析) / Translate (翻訳) / Lex (会話) / Textract (書類OCR)
  [3] 音声を扱う?
          YES --> Transcribe (音声→テキスト) / Polly (テキスト→音声)
  [4] 数値データの予測・推薦?
          YES --> Forecast (時系列予測) / Personalize (推薦)
  [5] 生成AI・LLMが必要?
          YES --> Amazon Bedrock
  [6] カスタムモデルが必要 or 上記に当てはまらない?
          YES --> Amazon SageMaker

  コスト最小化 + 管理不要 --> AI Services (1〜5) を優先
  高度なカスタマイズ + 独自データ --> SageMaker

試験頻出パターン まとめ

1

「カスタマイズ不要 vs カスタムモデル」

特定タスクに特化したAIサービス（Rekognition等）はコードなし・ML知識不要。独自ビジネスロジックやカスタムデータが必要な場合はSageMakerを選ぶ。

2

「生成AI = Amazon Bedrock」

テキスト生成・画像生成・コード生成・RAG・エージェント → すべてBedrock。マルチモデル（Claude・Titan・Stable Diffusion等）を選択できる点が特徴。

3

「医療AIは規制・PHI・説明可能性」

医療ユースケースでは精度だけでなく、HIPAA準拠・説明可能性（XAI）・FDA規制への対応が必須。HealthLake・Comprehend Medical を活用。

4

「製造業 = Lookout for Vision / Equipment」

外観検査 → Lookout for Vision（正常品画像のみで訓練可能）。設備異常検知 → Lookout for Equipment（教師なし）。どちらもML知識不要。

5

「ハルシネーション対策 = RAG + Guardrails」

LLMが誤情報を生成するリスクへの対策はRAG（Knowledge Bases）とBedrock Guardrailsの組み合わせ。出典明記で信頼性を担保。

演習問題（AIF-C01 スタイル）

問1. 大手ECサイトが「ユーザーの購買・閲覧履歴をもとにパーソナライズされた商品推薦」を最短時間で実装したいと考えています。機械学習の専門知識なしで実現できる最適なAWSサービスはどれですか？

A) Amazon SageMaker + 協調フィルタリングの自作モデル
B) Amazon Personalize ★正解★
C) Amazon Rekognition
D) Amazon Comprehend

解説: Amazon Personalize はAmazonが自社ECで使うのと同じ推薦アルゴリズムをマネージドサービスとして提供。購買・閲覧データをS3にアップロードするだけで実装可能。SageMakerは自作が必要で時間がかかる。Rekognitionは画像分析用。

問2. 製造工場でカメラを使った製品の外観検査を自動化したいです。不良品の画像は非常に少なく（数十枚程度）、正常品の画像は数百枚あります。最適なAWSサービスはどれですか？

A) Amazon Rekognition Image（カスタムラベル）
B) Amazon Lookout for Vision ★正解★
C) Amazon Textract
D) SageMaker Ground Truth

解説: Lookout for Vision は正常品の画像のみで訓練できる（教師なし異常検知）。不良品画像が少ない製造業の現実に対応。Rekognitionのカスタムラベルには多数のラベル付き画像が必要。Textractは文書OCR用。

問3. 金融機関が社内の規制・コンプライアンス文書（PDF500件）をもとに、従業員からの質問に正確に回答するシステムを構築したいです。回答の根拠（出典）も提示する必要があります。最適なアーキテクチャはどれですか？

A) Amazon Lex のみ
B) Amazon Bedrock Knowledge Bases（RAG構成） ★正解★
C) SageMaker + BERT のファインチューニング
D) Amazon Comprehend のみ

解説: RAG（Knowledge Bases）は社内文書をインデックス化し、質問に関連する文書チャンクをLLMに提供して回答を生成。出典URLも返せるのでハルシネーション対策に有効。Lexはインテントベースの会話AIで自由回答には不向き。ファインチューニングはコスト高で事実性の保証にもならない。

問4. 病院の電子カルテシステムで、医師の診察ノート（テキスト）から自動的に「疾患名」「処方薬」「検査値」を抽出し構造化データとして保存したいです。最適なAWSサービスはどれですか？

A) Amazon Comprehend（一般版）
B) Amazon Translate
C) Amazon Comprehend Medical ★正解★
D) Amazon Textract

解説: Amazon Comprehend Medical は医療テキストに特化したNLPサービス。疾患名・薬剤名・症状・検査値などの医療エンティティを自動抽出できる。一般版Comprehendでは医療用語を正確に処理できない。TextractはOCR（画像→テキスト）用。

問5. Amazon Bedrockで構築した社内チャットボットが、社内規定にない質問に対して誤った情報を自信満々に回答してしまう問題（ハルシネーション）が発生しています。最も効果的な対策はどれですか？

A) より高性能なLLMモデルに変更する
B) プロンプトに「正確に答えてください」と追記する
C) RAG（Knowledge Bases）を使い、社内文書のみを根拠として回答させる ★正解★
D) ファインチューニングで社内知識を学習させる

解説: ハルシネーション対策の最善策はRAG。社内文書のみをコンテキストとして与えることで、LLMが知らない情報は「情報がありません」と返せるようになる。高性能モデルへの変更やプロンプト修正では根本解決にならない。ファインチューニングはコスト高で事実性の保証にもならない。
