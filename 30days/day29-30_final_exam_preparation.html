2. 重要サービス比較まとめ

学習時間の目安: 40分
このセクションの目標: 試験で最も混乱しやすい「似たサービスの使い分け」を比較表で一気に整理する。各テーブルは試験直前の最終確認チートシートとして活用できる。

2.1 画像・動画分析: Rekognition vs SageMaker 画像分類

比較項目 Amazon Rekognition SageMaker 画像分類
主な用途 顔認識・物体検出・シーン検出・テキスト抽出・成人コンテンツ検出・有名人認識 独自カテゴリの画像分類（例: 自社製品の種別判定）
ML知識の必要性 不要（APIを呼ぶだけ） 必要（モデル選択・ハイパーパラメータ調整・訓練管理）
カスタムラベル Rekognition Custom Labels で独自ラベルを追加可能（少量の画像でOK） 完全カスタム（あらゆるラベルに対応）
推奨データ量 Custom Labels: カテゴリあたり10〜数百枚 カテゴリあたり数百〜数万枚
費用構造 API呼び出し回数課金 訓練時間 + エンドポイント稼働時間課金
選ぶべき状況 一般的な画像解析タスク・ML経験なし・すぐに使いたい 独自業務ドメインのカスタム分類・大量データ・高精度が必要
試験ポイント: 「カスタムモデル不要・迅速導入 → Rekognition」「独自データ・高度カスタマイズ → SageMaker」が基本判断軸。製造業の外観検査は Lookout for Vision（正常品画像のみで訓練可能な点が差別化）。

2.2 テキスト分析: Comprehend vs Textract vs Translate vs Transcribe

サービス 入力 出力・主な機能 代表的ユースケース
Amazon Comprehend テキスト（文字列） 感情分析・エンティティ抽出・キーフレーズ抽出・言語検出・トピックモデリング・カスタム分類 SNSの感情分析、カスタマーレビュー分析、契約書の分類
Amazon Textract 画像・PDF（書類スキャン） テキスト抽出（OCR）・フォーム解析（Key-Value）・テーブル抽出・署名検出 請求書の自動処理、身分証明書の読み取り、医療フォームのデジタル化
Amazon Translate テキスト（文字列） 75言語以上の機械翻訳・カスタム用語集対応・バッチ翻訳 グローバルECの商品説明翻訳、多言語カスタマーサポート
Amazon Transcribe 音声ファイル（MP3・WAV等） 音声→テキスト変換・話者分離・カスタム語彙・PII自動削除・医療特化版あり 会議の議事録自動化、コールセンター録音の分析、字幕生成
よく混同されるケース:
「書類の文字を読み取りたい」→ Textract（画像からOCR）
「文字から感情を分析したい」→ Comprehend（テキストNLP）
「話し言葉をテキストにしたい」→ Transcribe（音声認識）
「日本語を英語にしたい」→ Translate（機械翻訳）
Textractの出力をComprehendに渡すパイプラインがよく出題される。

2.3 音声・会話: Lex vs Polly vs Transcribe

サービス 変換方向 核心機能 ユースケース
Amazon Lex テキスト/音声 → 意図（Intent） 会話AI・チャットボット構築。インテント認識・スロット抽出・フルフィルメント（Lambda連携） 問い合わせ対応Bot、予約システム、Amazon Connect連携コールセンター
Amazon Polly テキスト → 音声 Neural TTS（自然な音声合成）・SSML対応・リアルタイム/バッチ変換・60以上の音声・多言語対応 ニュース読み上げ、アプリの音声ガイダンス、視覚障害者支援
Amazon Transcribe 音声 → テキスト 自動音声認識（ASR）・話者識別・カスタム語彙・PII削除・医療特化版 会議録音のテキスト化、コールセンター分析、字幕自動生成
覚え方: Lex = 「話しかける・意図を理解する」（会話AI）。Polly = 「しゃべる」（テキストを声に）。Transcribe = 「聞いて書く」（音声をテキストに）。Lexは内部でTranscribeを使って音声入力を処理している。

2.4 予測・異常検知: Forecast vs Lookout for Metrics vs Fraud Detector

サービス 何を検知/予測するか 主な特徴 代表的ユースケース
Amazon Forecast 時系列データの将来値を予測 DeepAR（Amazonオリジナル）・複数の統計・ML手法を自動選択・外部データ（天気等）の組み込み可能 需要予測・売上予測・在庫計画・エネルギー消費予測・Webトラフィック予測
Amazon Lookout for Metrics ビジネスKPIの異常を検知 正常範囲を自動学習・S3/CloudWatch/Redshift等と接続・異常の原因ディメンションも特定 売上の急落検知・広告CTRの異常・ECのコンバージョン率低下アラート
Amazon Fraud Detector 特定の不正・詐欺行為を検知 Amazonの不正検知ノウハウを活用・ルールベースとML併用・オンライン取引特化 オンライン支払い詐欺・新規アカウント不正登録・プロモーション乱用検知
よく混同されるケース:
「来月の売上を予測」→ Forecast（将来値の予測）
「今日の売上が急に落ちた → 原因を特定」→ Lookout for Metrics（KPI異常検知）
「クレジットカード不正取引を検知」→ Fraud Detector（不正行為検知）
「設備センサーの異常を検知」→ Lookout for Equipment（IoTデータ異常検知）

2.5 生成AI: Bedrock vs SageMaker JumpStart

比較項目 Amazon Bedrock SageMaker JumpStart
提供形態 フルマネージド（インフラ管理不要） SageMakerのワンクリックデプロイ機能
対応モデル Claude（Anthropic）・Titan・Stable Diffusion・Llama・Mistral 等の商用モデル Hugging Faceモデル含む数百のオープンソースモデル
カスタマイズ方法 Continued Pre-training・Fine-tuning・RAG（Knowledge Bases） ファインチューニング・SageMaker Pipelines連携
安全機能 Guardrails（コンテンツフィルター・PII削除・ハルシネーション検出） SageMaker標準のセキュリティ（IAM・VPC・KMS）
API形式 InvokeModel API（統一インターフェース） SageMaker エンドポイント API
選ぶべき状況 商用グレードの生成AI・RAGシステム・エージェント構築・すぐに使いたい オープンソースモデルを活用したい・完全なインフラ制御が必要・VPC内に閉じた環境が必要

2.6 データ準備: Glue vs SageMaker Data Wrangler vs SageMaker Processing

サービス 主な目的 UI・操作方法 スケール・特徴
AWS Glue ETL（Extract・Transform・Load）。データレイクへのデータ収集・変換・カタログ化 コード（PySpark・Python）またはGlue Studio（GUIのビジュアルETL） 大規模データ対応・サーバーレス・Athena/Redshift/S3との統合が強力
SageMaker Data Wrangler ML向けデータ準備・特徴量エンジニアリング・可視化・品質分析 GUI（ノーコード）中心。S3・Redshift・Athena等から取り込み ML特化（300以上の変換テンプレート）・Feature Storeへのエクスポート対応
SageMaker Processing 前処理・特徴量計算・評価スクリプトの大規模実行 コード（Python/Spark）。ProcessingJobとして実行 マネージドクラスターで大規模並列処理・Training Jobと連携したパイプラインに組み込み可
選択の目安: データエンジニアリング（ETL・データレイク構築）→ Glue。ML担当者がGUIでデータ探索・前処理 → Data Wrangler。本番MLパイプラインでの前処理スクリプト実行 → SageMaker Processing。

3.1 よく出るシナリオ問題のパターン

試験頻出シナリオ パターン 10選

1

「ML知識なし・すぐに使いたい」パターン

問題文に「機械学習の専門知識なし」「短期間で」「コードなし」というキーワードがあれば AWSのマネージドAIサービス（Rekognition・Comprehend・Forecast等） が正解。SageMakerは知識が必要。

2

「独自データ・カスタムモデル」パターン

「独自の分類カテゴリ」「既存サービスでは対応できない業界固有の分析」「大量の自社データ」というキーワードがあれば Amazon SageMaker が正解。

3

「ハルシネーション対策」パターン

「LLMが誤情報を生成する問題」「回答の根拠・出典が必要」「社内文書に基づいた正確な回答」というキーワードがあれば RAG（Amazon Bedrock Knowledge Bases） が正解。ファインチューニングはこの問題を解決しない。

4

「コスト最適化」パターン

「推論コストを削減したい」「常時稼働は不要」というキーワードがあれば SageMakerサーバーレス推論 または バッチ変換（Batch Transform）。リアルタイムエンドポイントは24時間課金されるため高コスト。

5

「時系列データの将来予測」パターン

「在庫の最適化」「需要予測」「売上予測」というキーワードがあれば Amazon Forecast。SageMakerのDeepAR組み込みアルゴリズムも時系列予測に使えるが、マネージドで簡単なのはForecast。

6

「モデルの公平性・バイアス」パターン

「差別的な判断を防ぐ」「性別・人種での偏り検出」「説明可能なAI」というキーワードがあれば SageMaker Clarify。バイアス検出レポートとSHAP値による説明可能性分析を提供。

7

「モデルの品質劣化を検知」パターン

「本番デプロイ後の精度低下を自動検知」「データドリフトの監視」というキーワードがあれば SageMaker Model Monitor。CloudWatch と連携してアラートを自動送信。

8

「製造ラインの外観検査」パターン

「正常品画像しかない」「不良品が少ない」「製造ラインの検査自動化」というキーワードがあれば Amazon Lookout for Vision。正常品のみで訓練できる点が決め手。

9

「複数ステップの自律タスク実行」パターン

「ユーザーの複雑な依頼を自動で分解・実行」「APIを自律的に呼び出す」というキーワードがあれば Amazon Bedrock エージェント（ReActパターン）。

10

「段階的なデプロイとロールバック」パターン

「新モデルをリスクなく本番導入したい」「問題時は素早く旧バージョンに戻したい」というキーワードがあれば Blue/Greenデプロイ または SageMakerカナリアデプロイ。

3.2 紛らわしいサービスの違い（ひっかけ問題対策）

混乱しやすいペア 正しい使い分け
Rekognition vs Textract Rekognitionは「画像の中に何が映っているか」を分析（人・物体・感情・有名人）。TextractはOCR = 「画像の中に書いてある文字を読み取る」。
Transcribe vs Lex Transcribeは「音声をテキストに書き起こす」だけ。Lexは「テキスト/音声の意図を理解してボットとして応答する」会話AIエンジン。
Comprehend vs Comprehend Medical 一般版Comprehendは汎用テキスト分析。Medical版は「疾患名・薬剤名・副作用・ICD-10コード」など医療特有エンティティを正確に抽出できる。
Forecast vs Lookout for Metrics Forecastは「未来の値を予測する（将来値の推定）」。Lookout for Metricsは「現在進行中の異常を検知する（現在の異常検出）」。目的が逆。
Fine-tuning vs RAG Fine-tuningはモデルの重みを更新して新しい知識を学習（コスト高・時間かかる）。RAGはモデルを変えずに外部文書を参照（コスト低・すぐに使える・出典明記可）。ハルシネーション対策はRAG。
SageMaker Autopilot vs AutoML（手動） Autopilotは完全自動でデータを渡すだけで最良のモデルを選定・訓練（プロセスの透明性あり＝各手法のノートブックを確認可能）。
Lookout for Vision vs Lookout for Equipment Vision = 「カメラ画像で製品の外観異常を検知」（製造ライン検査）。Equipment = 「センサーデータ（振動・温度）で設備の異常を検知」（予知保全）。
SageMaker Processing vs Glue Processing はMLパイプライン内の前処理スクリプト実行（ML特化）。Glueは汎用ETL・データカタログ（データエンジニアリング全般）。

4.3 評価指標（Precision・Recall・F1・AUC-ROC）

混同行列の基本要素:

TP = True Positive （正解: 陽性 → 予測: 陽性）正しく陽性と判定
FP = False Positive （正解: 陰性 → 予測: 陽性）誤って陽性と判定（誤報）
FN = False Negative （正解: 陽性 → 予測: 陰性）誤って陰性と判定（見逃し）
TN = True Negative （正解: 陰性 → 予測: 陰性）正しく陰性と判定

Precision（適合率）= TP / (TP + FP)　← 「陽性と言ったうち本当に陽性の割合」
Recall（再現率）= TP / (TP + FN)　← 「本当の陽性のうち正しく見つけた割合」
F1 Score = 2 × Precision × Recall / (Precision + Recall)　← 調和平均
Accuracy = (TP + TN) / (TP+FP+FN+TN)　← 不均衡データでは使えない
状況 重視すべき指標 理由
がん診断・詐欺検知 Recall（再現率）優先 見逃し（FN）のコストが大きい。多少の誤報（FP）は許容。
スパムフィルター・コンテンツ審査 Precision（適合率）優先 正常メールをスパムと誤判定（FP）すると重要メールを失う。見逃し（FN）は許容。
バランスを取りたい場合 F1スコア PrecisionとRecallのトレードオフを一つの指標で表す。不均衡データでも有効。
閾値の影響を排除して評価 AUC-ROC あらゆる閾値設定でのモデルの総合的な識別能力を表す。1.0が完璧・0.5がランダム。

4.4 MLOps用語

用語 意味・ポイント
Data Drift データドリフト。本番環境に入ってくる入力データの統計的性質（分布）が、訓練時と変化すること。例: コロナ前に訓練した需要予測モデルがコロナ後に精度低下。
Concept Drift コンセプトドリフト。入力データと出力（正解）の関係性そのものが変化すること。例: 詐欺師が手口を変えることで以前の詐欺検知モデルが機能しなくなる。
CI/CD for ML 継続的インテグレーション/継続的デリバリーのML版。コード変更→自動テスト→自動訓練→自動デプロイのパイプラインを整備すること。SageMaker Pipelines + CodePipeline。
Shadow Mode シャドウモード。本番トラフィックをリアルタイムに新モデルにも送って予測させるが、ユーザーへの応答は旧モデルのみ使用。新モデルの品質を実績データで検証する手法。
Feature Store 特徴量ストア。前処理済みの特徴量を再利用可能な形で集中管理するリポジトリ。SageMaker Feature StoreはオンラインストアとオフラインストアをAPI一つで管理。
Model Lineage モデル系譜。あるモデルがどのデータ・コード・ハイパーパラメータで生成されたかを追跡記録すること。SageMaker ML LineageとExperimentsで管理。

時間超過の罠: 1問に5分以上かけると残りの問題を解く時間が足りなくなります。わからない問題は30秒考えて見当がつかなければ即フラグ → 次の問題へ進む。「知識があれば解ける問題」を確実に取ることが優先です。

5.2 問題の解き方のコツ

消去法を徹底活用する

「明らかに違う選択肢」を先に消去する。4択なら2つ消して残り2つから選ぶ確率は50%。完全にわからなくても消去法で正答率は上がる。

消去の基準: 問われていないサービス・逆の効果・コスト的に非現実的・「常に」「絶対に」などの極端な表現。

キーワードに注目する

問題文中の特定のキーワードが正解を示す手がかりになる。

  • 「ML知識不要」→ マネージドAIサービス
  • 「カスタムモデル」「独自データ」→ SageMaker
  • 「生成AI」「LLM」「テキスト生成」→ Bedrock
  • 「時系列」「将来予測」→ Forecast
  • 「ハルシネーション」「出典」→ RAG

「最も適切な」に注意する

AWSの試験は「間違いを選べ」ではなく「最も適切なものを選べ」という形式が多い。複数の選択肢が「正しく見える」ことがある。

判断基準: (1) 要件を全て満たすか (2) よりシンプルか (3) コスト効率が良いか (4) AWS推奨アーキテクチャに近いか

複数選択問題の攻略

「2つ選べ」「3つ選べ」の問題は部分点なし（全て正解しないと点数にならない）。

戦略: まず確実に正しい選択肢を見つけ、残りをその観点から評価する。迷う場合は「コスト最小」「AWSベストプラクティス」を基準に選ぶ。

5.3 わからない問題への対処法

「30秒ルール」:
問題を読んで30秒以内に方向性が見えない場合は、その場で考え込まず以下のステップで対処する。
  1. 最も「ありえない」選択肢を1〜2つ消去する
  2. 残った選択肢の中で「AWSらしい」（マネージド・セキュア・スケーラブル）選択肢を選ぶ
  3. 問題にフラグを立てて次の問題へ移る
  4. 全問を一回解いた後で戻って再検討する
「AWSらしい」回答の特徴:
AWSの試験では、セキュリティ・スケーラビリティ・コスト効率・マネージドサービスの活用を優先する回答が正解になりやすい。「自前でサーバーを管理する」「手動でスケールする」「カスタムコードを書く」より「マネージドサービスを使う」「自動スケール」「サーバーレス」の選択肢を優先する。

5.4 見直しのポイント

  • マーク忘れの確認: 全問題に回答済みであることを確認（未回答問題は確実に0点）
  • 読み間違いの確認: 「〜でない」「〜以外」などの否定形を見落としていないか
  • 複数選択の個数確認: 「2つ選べ」の問題で1つしか選んでいないケースはないか
  • 第一印象を信じる: 確信なく変更した答えが不正解になることが多い。再検討時は「なぜ変更するのか明確な理由があるか」を問う
  • 時間切れ対策: 残り5分でフラグ問題が残っている場合は「ランダムに選ぶより根拠のある推測」で回答する

B) Amazon Textract
C) Amazon Comprehend Medical ★正解★

解説: Comprehend Medicalは医療テキストから疾患名・薬剤名・副作用などの医療固有エンティティを抽出する専用サービス。一般版Comprehendでは医療用語を正確に処理できない。TextractはOCR（画像からテキスト変換）。TranscribeはASR（音声→テキスト）。

問2. 機械学習モデルを本番環境にデプロイした後、入力データの統計的分布が訓練時と大きく変化したことを自動で検知し、アラートを出したいです。最適なAWSサービスはどれですか？

A) AWS CloudTrail
B) Amazon SageMaker Model Monitor ★正解★
C) Amazon CloudWatch Logs
D) Amazon SageMaker Clarify

解説: SageMaker Model Monitorはデータ品質・モデル品質・バイアス・特徴量属性の4種類を継続的に監視し、CloudWatchと連携してアラートを発行する。CloudTrailはAPI操作ログ。Clarifyはバイアス検出と説明可能性分析（定期実行）。

問3. 企業が新しい機械学習モデルを本番環境にリリースする際、問題発生時に即座に旧モデルに切り戻せるようにしながら、徐々にトラフィックを新モデルに移行したいです。最適なデプロイ戦略はどれですか？

A) SageMaker バッチ変換（Batch Transform）
B) SageMaker カナリアデプロイ（段階的トラフィック移行） ★正解★
C) SageMaker サーバーレス推論
D) SageMaker Autopilot

解説: カナリアデプロイはトラフィックを10%→50%→100%のように段階的に新バージョンへ移行し、問題があれば即座にロールバックできる。バッチ変換は非リアルタイムの一括推論。サーバーレス推論は低頻度推論のコスト最適化。Autopilotは自動ML。

問4. Amazon BedrockのLLMを使ったチャットボットで、ユーザーが競合他社の製品について質問した場合に自動的に「回答できません」と返し、個人情報（クレジットカード番号等）が入力された場合はマスキングしたいです。最適なBedrockの機能はどれですか？

A) Amazon Bedrock Knowledge Bases
B) Amazon Bedrock Guardrails ★正解★
C) Amazon Bedrock Agents
D) SageMaker Clarify

解説: Bedrock Guardrailsは（1）禁止トピックのフィルタリング（2）センシティブ情報のマスキング（3）有害コンテンツのブロック（4）ハルシネーション検出を提供するコンテンツフィルタリング機能。Knowledge Basesはデータソース参照（RAG）。Agentsは自律的なタスク実行。

問5. 不均衡なデータセット（陽性クラスが全体の1%）で訓練した二値分類モデルを評価したいです。このケースで最も適切な評価指標はどれですか？

A) Accuracy（正解率）
B) AUC-ROC ★正解★
C) Mean Squared Error（MSE）
D) R² スコア

解説: 不均衡データでAccuracyを使うと「全部陰性と予測」するだけで99%の正解率が出てしまい意味がない。AUC-ROCは閾値に依存せずモデルの識別能力を評価できる。MSEとR²は回帰問題の指標。不均衡データにはF1スコアやAUC-PRも有効。
