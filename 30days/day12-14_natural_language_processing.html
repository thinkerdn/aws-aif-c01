1. NLPの概要と歴史

⏱ 学習時間の目安：20分

1.1 自然言語処理とは

自然言語処理（Natural Language Processing：NLP）は、人間が日常的に使う言語（自然言語）をコンピューターが理解・生成・操作できるようにするAIの一分野です。 テキストや音声の形で表現された言語データを扱い、意味の理解・情報抽出・翻訳・生成など幅広いタスクをカバーします。

ポイント： NLPは「言語の理解（NLU：Natural Language Understanding）」と「言語の生成（NLG：Natural Language Generation）」の2つの側面を持ちます。 現代のLLM（大規模言語モデル）はこの両方を統合しています。

1.2 NLPが難しい理由

自然言語はプログラミング言語と異なり、以下の特性により機械処理が困難です：

  • 曖昧性（Ambiguity）：「彼は眼鏡の男を追いかけた」→ 眼鏡をかけているのは彼か男か？
  • 文脈依存性：「銀行に行く」→ 金融機関か川の土手か？
  • 慣用表現・比喩：「猫の手も借りたい」→ 文字通りではない意味
  • 言語の多様性：世界に7,000以上の言語。文法構造が大きく異なる
  • 進化する言語：スラング、新語、略語が常に生まれる

1.3 NLPの歴史的発展

1

1950〜1980年代：ルールベース時代

人間が手動で文法規則や辞書を作成。チューリングテスト提案（1950）、ELIZAチャットボット（1966）。処理できる範囲が限定的で拡張性に乏しかった。

2

1990〜2000年代：統計的NLP時代

大量のテキストコーパスから確率モデルを学習。HMM（隠れマルコフモデル）、Naive Bayes分類器、TF-IDFなど。機械翻訳や音声認識に応用。

3

2010〜2017年：ニューラルNLP時代

Word2Vec（2013）による単語埋め込みが登場。RNN・LSTM・GRUによるシーケンスモデリング。Attention機構の発明（2015）。精度が大幅向上。

4

2017年〜現在：Transformer・LLM時代

「Attention Is All You Need」論文（2017）でTransformer登場。BERT（2018）、GPT系列、T5、LLaMAなど大規模言語モデルが次々登場。ChatGPT（2022）で一般普及。

具体例：機械翻訳精度の進化
ルールベース：「I have a book」→ 「私は本を持っている」（単純な文は正確だが複雑な文は誤訳多発）
統計的：フレーズ対応を学習、自然さが向上するが流暢さに課題
ニューラル（Transformer）：文脈全体を考慮した自然な翻訳。Google翻訳・DeepL等で実用化

2. テキスト前処理の基礎

⏱ 学習時間の目安：25分

NLPモデルに生のテキストをそのまま入力することは稀です。多くの場合、テキストを数値表現に変換するための前処理が必要です。

2.1 トークン化（Tokenization）

テキストを処理単位（トークン）に分割する操作です。

単語トークン化

「I love NLP」→ ["I", "love", "NLP"]

スペースや句読点で分割。英語には有効だが日本語には不向き。

サブワードトークン化

「unhappiness」→ ["un", "happy", "ness"]

BPE（Byte Pair Encoding）やWordPieceが代表的。LLMで広く使用。

文字トークン化

「NLP」→ ["N", "L", "P"]

語彙外単語（OOV）問題を完全回避できるが、シーケンスが長くなる。

日本語トークン化

MeCab・SudachiなどのMorphological Analyzerを使用。
「東京に行く」→ ["東京", "に", "行く"]

2.2 テキスト正規化

  • 小文字化：「Apple」「apple」「APPLE」を統一
  • ステミング（Stemming）：「running」「runs」「ran」→「run」（語根抽出、ルールベース）
  • レンマ化（Lemmatization）：「better」→「good」（辞書を使った正確な原形変換）
  • ストップワード除去：「the」「is」「a」など頻出するが意味を持たない語を削除
  • 特殊文字・HTMLタグ除去：ノイズとなる記号やマークアップを削除
# Pythonによるテキスト前処理の例（概念コード） import re text = "The running dogs RAN quickly!!! Visit https://example.com" # 1. 小文字化 text = text.lower() # → "the running dogs ran quickly!!! visit https://example.com" # 2. URLの除去 text = re.sub(r'https?://\S+', '', text) # → "the running dogs ran quickly!!! visit " # 3. 特殊文字除去 text = re.sub(r'[^a-z\s]', '', text) # → "the running dogs ran quickly visit " # 4. ストップワード除去（"the", "a", "is" など） stopwords = {"the", "a", "is", "visit"} tokens = [w for w in text.split() if w not in stopwords] # → ["running", "dogs", "ran", "quickly"]

2.3 テキストの数値表現

機械学習モデルはテキストをそのまま処理できないため、数値ベクトルに変換します。

手法 概要 特徴 用途
Bag of Words (BoW) 単語の出現頻度をベクトル化 シンプル、語順情報なし 文書分類、スパム検出
TF-IDF 単語の重要度を頻度と希少性で重み付け BoWより高精度、次元数大 情報検索、キーワード抽出
Word2Vec 単語を密なベクトル空間に埋め込み 意味的類似性を表現できる 感情分析、文書類似度
BERT Embeddings 文脈を考慮した動的埋め込み 高精度、計算コスト大 質問応答、固有表現認識

Word2Vecの直感的理解

Word2Vecは単語を300次元程度のベクトルとして表現します。重要な性質：

  • 意味が近い単語はベクトル空間で近くに配置される
  • ベクトル演算が意味的演算に対応：
    vector("王") - vector("男") + vector("女") ≈ vector("女王")
  • 類義語・反義語・カテゴリ関係などを自動的に学習

3. 従来のNLP手法（RNN・LSTM）

⏱ 学習時間の目安：20分

3.1 統計的言語モデル（n-gram）

n-gramモデルは最も基本的な言語モデルです。直前のn-1個の単語から次の単語を予測します。

Bigram（2-gram）モデルの例 文：「東京は日本の首都です」 P("東京は日本の首都です") ≈ P("東京") × P("は"|"東京") × P("日本"|"は") × P("の"|"日本") × P("首都"|"の") × P("です"|"首都") → 各条件付き確率をコーパスから計算してモデルを構築
n-gramの限界： n=2や3の場合、長距離の依存関係を捉えられません。 「私が昨日公園で出会った彼は...優しかった」→「彼」と「優しかった」の関係を捉えるには長い文脈が必要。

3.2 RNNとLSTM

リカレントニューラルネットワーク（RNN）は、シーケンスデータを処理するために設計されました。 前のステップの隠れ状態を次のステップに引き継ぐことで、文脈情報を保持します。

RNNの処理フロー 入力系列： x₁ x₂ x₃ x₄ ↓ ↓ ↓ ↓ 隠れ状態： h₁ → h₂ → h₃ → h₄ ↓ ↓ ↓ ↓ 出力： y₁ y₂ y₃ y₄ 各ステップ：h_t = tanh(W_h × h_{t-1} + W_x × x_t + b)

RNNの問題点

  • 勾配消失問題：長い系列で初期の情報が失われる
  • 逐次処理：並列化できず学習が遅い
  • 長期依存性：遠く離れた単語の関係を学習しにくい

LSTM/GRUによる改善

  • ゲート機構：入力・忘却・出力ゲートで情報を選択的に保持
  • 長期記憶：RNNより長い依存関係を学習可能
  • GRU：LSTMを簡略化したモデル、計算効率が良い

3.3 Seq2Seqモデル

機械翻訳などの「系列→系列」変換タスクのために、Encoder-Decoderアーキテクチャが提案されました。

Seq2Seqの構造（機械翻訳の例） 入力（英語）： I love NLP ↓ ↓ ↓ Encoder（LSTM）: → → → → → → → コンテキストベクトル c ↓ Decoder（LSTM）: c → "私" → "は" → "NLPが" → "好きです" 出力（日本語）：私はNLPが好きです
Seq2Seqのボトルネック： Encoderの出力が固定長のコンテキストベクトル1つのため、 長い文章の情報を全て圧縮する必要があり、翻訳精度が低下します。 これを解決するためにAttention機構が登場しました。

4. Attention機構

⏱ 学習時間の目安：25分

4.1 Attention機構とは

Attention（注意機構）は、Decoderが出力を生成する際にEncoder内のどの入力トークンに「注目」すべきかを学習する仕組みです。 固定長のボトルネックを解消し、翻訳精度を大幅に向上させました。

直感的な理解：
英語「The cat sat on the mat」を日本語に翻訳する際：
「猫」を出力するとき → 「cat」に高いAttentionスコア
「座った」を出力するとき → 「sat」に高いAttentionスコア
「マットの上に」を出力するとき → 「on」「mat」に高いAttentionスコア
→ モデルが翻訳時にどの原文単語を参照するかを動的に決定

4.2 Self-Attention（自己注意機構）

Transformerで中心となる概念です。同じ系列内のすべてのトークン間の関係を同時に計算します。

Self-Attentionの概念 文：「彼女はスミスさんが来ると知っていた」 "彼女" が注目するトークン（Attentionスコア）： 彼女 ←→ は (0.05) 彼女 ←→ スミスさん (0.60) ← 高い注意 彼女 ←→ が (0.10) 彼女 ←→ 来る (0.05) 彼女 ←→ 知っていた (0.20) → "彼女"と"スミスさん"の関係が強いことを表現 → 代名詞解析（"彼女"が誰を指すか）などに有効 計算式：Attention(Q, K, V) = softmax(QK^T / √d_k) × V Q（Query）、K（Key）、V（Value）はそれぞれ学習された線形変換

4.3 Multi-Head Attention

複数のAttentionヘッドを並列に実行することで、異なる種類の依存関係（文法的・意味的・位置的）を同時に学習できます。

Attentionヘッド 学習する関係の例
ヘッド1 主語-述語の対応
ヘッド2 代名詞と指示対象の対応
ヘッド3 動詞と目的語の対応
ヘッド4〜N その他の文法・意味的関係

5. Transformerアーキテクチャ

⏱ 学習時間の目安：25分

2017年のGoogle論文「Attention Is All You Need」で提案されたTransformerは、RNNを一切使わずSelf-Attentionのみで構成された革命的なアーキテクチャです。

5.1 Transformerの構造

Transformerの全体構造 入力テキスト ↓ [トークン化 + 埋め込み] ↓ [位置エンコーディング] ↓ ┌──────────────────┐ │ Encoderブロック ×N │ │ ┌──────────────┐ │ │ │ Multi-Head │ │ │ │ Attention │ │ │ └──────┬───────┘ │ │ Add & LayerNorm │ │ ┌──────────────┐ │ │ │ Feed-Forward │ │ │ │ Network │ │ │ └──────┬───────┘ │ │ Add & LayerNorm │ └──────────────────┘ ↓ ┌──────────────────┐ │ Decoderブロック ×N │ │ Masked Self-Att │ │ Cross-Attention │ │ Feed-Forward │ └──────────────────┘ ↓ [線形層 + Softmax] ↓ 出力テキスト

5.2 主要コンポーネント

位置エンコーディング（Positional Encoding）

Transformerは全トークンを並列処理するため、語順情報がありません。 位置エンコーディングは各トークンの位置情報をベクトルとして付与します。 sin/cos関数を使った固定エンコーディング（元のTransformer）や、学習可能な位置埋め込みが使われます。

残差接続（Residual Connection）と層正規化

各サブレイヤーの前後で「入力をそのまま足す」残差接続と、 層正規化（Layer Normalization）を組み合わせることで、深いネットワークの安定した学習を実現します。
出力 = LayerNorm(x + SubLayer(x))

Feed-Forward Network（FFN）

各Attentionブロックの後に配置される全結合ネットワーク。 各位置のトークンに独立して適用されます。 Transformerの「知識ストレージ」として機能し、事実情報を蓄えるとも解釈されています。

5.3 Transformerのバリエーション

タイプ 代表モデル アーキテクチャ 主な用途
Encoder-only BERT, RoBERTa 双方向Attention（全文脈参照） 分類、固有表現認識、質問応答
Decoder-only GPT-4, LLaMA, Claude 単方向Attention（左→右） テキスト生成、対話、コード生成
Encoder-Decoder T5, BART 両方を組み合わせ 翻訳、要約、Q&A
試験頻出： BERTは双方向（Bidirectional）Transformerであり、 テキストの左右両方の文脈を参照してトークンを処理します。 GPT系は単方向（Autoregressive）で、左から右へ順番にトークンを生成します。

6. 大規模言語モデル（LLM）

⏱ 学習時間の目安：30分

6.1 LLMとは

大規模言語モデル（Large Language Model：LLM）は、数十億〜数千億のパラメータを持ち、 インターネット規模のテキストデータで事前学習されたTransformerモデルです。 汎用的な言語理解・生成能力を持ち、様々なタスクにファインチューニングや プロンプトエンジニアリングで適用できます。

6.2 事前学習と転移学習

LLMの学習パイプライン ステップ1：事前学習（Pre-training） 大量テキストデータ（Common Crawl, Wikipedia等、数TB〜数百TB） ↓ 自己教師あり学習（次トークン予測 or マスクトークン予測） ↓ 基盤モデル（Foundation Model）完成 ステップ2：ファインチューニング（Fine-tuning） タスク固有データ（数千〜数十万サンプル） ↓ 特定タスク向けに重みを調整 ↓ タスク特化モデル完成 ステップ3（オプション）：RLHF 人間によるフィードバック ↓ 強化学習で有用性・安全性を向上 ↓ ChatGPT, Claude等の対話モデル完成

6.3 プロンプトエンジニアリング

LLMの能力を引き出すためのプロンプト（入力文）の設計技術です。

手法 説明 例
Zero-shot 例示なしで直接指示 「次の文を翻訳してください：I love NLP」
Few-shot 数例を示してから指示 「例1：... 例2：... ではこれは？」
Chain-of-Thought 推論ステップを段階的に記述させる 「ステップバイステップで考えてください」
Role Prompting 特定の役割を演じさせる 「あなたは熟練した医師です...」
RAG（検索拡張生成） 外部知識を検索してプロンプトに追加 最新情報・社内文書への対応に有効

6.4 ファインチューニング手法の比較

手法 説明 コスト 適用場面
Full Fine-tuning 全パラメータを更新 非常に高い タスクが大きく異なる場合
LoRA 低ランク行列を追加して少数パラメータを更新 中程度 資源制約がある場合の主流手法
プロンプトチューニング ソフトプロンプトのみを学習 低い モデルを変更したくない場合
RLHF 人間フィードバックで強化学習 非常に高い 対話モデルの安全性・有用性向上

6.5 LLMの評価指標

Perplexity（困惑度）

言語モデルがテキストをどれだけ「驚く」かを測定。
低いほど良い。テキスト生成タスクで使用。

BLEU スコア

機械翻訳の品質評価。参照訳とのn-gram一致度を測定。
0〜1（高いほど良い）。

ROUGE スコア

要約の品質評価。参照要約との単語・フレーズ一致度。
ROUGE-N, ROUGE-Lなど種類がある。

Human Evaluation

人間が流暢性・正確性・有用性などを評価。
最も信頼性が高いが、コストと時間がかかる。

7. 主要なNLPタスク

⏱ 学習時間の目安：20分

7.1 テキスト分類

テキストをカテゴリに分類するタスクです。

  • 感情分析（Sentiment Analysis）：「このレストランは最高！」→ Positive
  • スパム検出：メールがスパムか否かを判定
  • トピック分類：ニュース記事を「政治・経済・スポーツ」に分類
  • 意図分類（Intent Classification）：チャットボットでユーザーの意図を判定

7.2 情報抽出

固有表現認識（NER）

テキストから固有名詞（人名・地名・組織名・日時等）を抽出。
「田中太郎は東京大学で2024年4月に研究を開始した」
→ 人名・組織名・日時をそれぞれ識別・抽出

関係抽出（Relation Extraction）

エンティティ間の関係を抽出。
「イーロン・マスクはテスラのCEOである」
→ (イーロン・マスク, CEO, テスラ)

7.3 機械翻訳と要約

機械翻訳（Machine Translation）

ある言語のテキストを別の言語に変換します。Google翻訳やDeepLが代表例。 現代の翻訳システムはTransformerベースのニューラル機械翻訳（NMT）を使用。 評価指標：BLEUスコア

テキスト要約（Text Summarization）

  • 抽出型（Extractive）：元の文章から重要な文をそのまま抽出して組み合わせる。シンプルで信頼性高い。
  • 生成型（Abstractive）：内容を理解して新しい文章として要約。自然だがハルシネーションのリスクあり。

評価指標：ROUGEスコア

7.4 質問応答と情報検索

タスク 入力 出力 手法例
抽出型QA 文書 + 質問 文書内の回答スパン BERT、SQuADデータセット
生成型QA 質問（＋文書） 生成された回答文 GPT系LLM、T5
RAG 質問 検索結果を基にした回答 Vector DB + LLM

7.5 ハルシネーション（幻覚）問題

重要：ハルシネーション（Hallucination）とは
LLMが事実と異なる情報を自信を持って生成してしまう現象です。
  • 存在しない論文や著者名を引用する
  • 実在しない法律や判例を挙げる
  • 計算結果を誤って示す
対策： RAGによる外部知識の活用、出力の事実確認、 temperature設定の最適化、Chain-of-Thoughtによる推論ステップの明示化

  • バッチ翻訳：S3の大量ドキュメントを一括翻訳

8.3 Amazon Lex

音声・テキスト対話インターフェース（チャットボット・音声ボット）構築サービスです。 AlexaのNLUエンジンを使用しています。

主要コンポーネント

  • Intent（意図）：ユーザーが何をしたいか（例：フライト予約）
  • Utterances（発話例）：Intentに対応するフレーズ
  • Slot（スロット）：パラメータ（出発地、日時等）
  • Fulfillment：Lambdaでビジネスロジックを実行

統合サービス

  • Amazon Connect（コールセンター）
  • Slack / Facebook Messenger
  • Twilio（SMS/音声）
  • AWS Lambda（バックエンド処理）

8.4 Amazon Textract

OCR（光学文字認識）を超えた、ドキュメント分析サービスです。スキャン文書やPDFからテキスト・表・フォームを抽出します。

ComprehendとTextractの使い分け：
Textract = 画像・PDFからテキストを抽出（OCR相当）
Comprehend = テキストの意味を分析（分類・感情・NER等）
→ 組み合わせて使うことが多い（Textractで抽出 → Comprehendで分析）

8.5 Amazon Kendra

機械学習ベースのエンタープライズ検索サービスです。自然言語クエリで社内文書を検索できます。 S3、SharePoint、Salesforceなど40以上のデータソースに接続可能です。

8.6 Amazon Bedrock と生成AI

フルマネージドの生成AIサービスで、Anthropic (Claude)、Meta (Llama)、AI21 Labs、Stability AIなど 複数のFMにAPIでアクセスできます。

機能 説明
Knowledge Bases for Bedrock RAGのフルマネージド実装。S3の文書をベクトル化して検索可能にする
Agents for Bedrock 複数ステップのタスクを自律的に実行するAIエージェントを構築
Model Customization Fine-tuning、Continued Pre-trainingによるモデルカスタマイズ
Guardrails 有害コンテンツのフィルタリング、PIIの検出・マスキング

8.7 その他のAWS NLP関連サービス

サービス 説明 主なユースケース
Amazon Polly テキスト読み上げ（TTS） 音声アシスタント、コンテンツ音声化
Amazon Transcribe 音声認識（ASR）、音声→テキスト変換 会議録音の文字起こし、字幕生成
Amazon Q 企業向け生成AIアシスタント 社内知識検索、コード支援（Amazon Q Developer）

9. 試験対策ポイント

⏱ 学習時間の目安：15分

9.1 シナリオ別 正解サービス早見表

シナリオ 正解サービス/手法
レビューの感情分析（ポジティブ/ネガティブ） Amazon Comprehend（感情分析機能）
コールセンター向け音声チャットボット Amazon Lex + Amazon Connect
医療記録から病名・薬名を抽出 Amazon Comprehend Medical
PDFフォームからデータを構造化抽出 Amazon Textract
社内ドキュメントへの自然言語検索 Amazon Kendra
多言語コンテンツをリアルタイム翻訳 Amazon Translate
会議音声を自動文字起こし Amazon Transcribe
フルマネージドでClaude/Llamaを使いたい Amazon Bedrock
社内文書を参照して正確な回答を生成（幻覚対策） RAG + Knowledge Bases for Bedrock
特定ドメインのモデルカスタマイズ（低コスト） LoRA / Fine-tuning

9.2 紛らわしいサービスの区別

Comprehend vs Kendra vs Bedrock

  • Comprehend：テキストの分析・分類・抽出（感情、NER、言語検出）。構造化された分析結果を返す。
  • Kendra：ドキュメントの検索。自然言語クエリで関連文書を検索・ランキング。
  • Bedrock：生成AIによるテキスト生成・対話。FMへのアクセス基盤。

Lex vs Transcribe vs Polly

  • Lex：対話管理（Intent理解 → 応答フロー制御）。チャットボットの「脳」。
  • Transcribe：音声→テキスト変換のみ（文字起こし）。
  • Polly：テキスト→音声変換のみ（読み上げ）。

9.3 重要な概念の整理

Transformerのポイント：
  • RNNを使わずSelf-Attentionのみで構成
  • 並列処理が可能で学習が高速
  • 長距離依存関係を効果的に捉えられる
  • 「Attention Is All You Need」（2017, Google）が起源
BERTのポイント：
  • 双方向Transformer（Bidirectional Encoder Representations from Transformers）
  • Masked Language Modeling（MLM）とNext Sentence Prediction（NSP）で事前学習
  • 分類・NER・QAなど理解タスクに優れる
  • テキスト生成には不向き（双方向なのでAutoregressive生成できない）
GPTのポイント：
  • 単方向Transformer（左→右にAutoregressiveに生成）
  • 次のトークン予測で事前学習
  • テキスト生成・対話に優れる
  • Few-shot/Zero-shot学習能力が高い
Day 12-14 学習チェックリスト：
  • □ トークン化・テキスト前処理の手順を説明できる
  • □ BoW・TF-IDF・Word2Vecの違いを説明できる
  • □ RNNの問題点とTransformerが解決した点を説明できる
  • □ Self-Attentionの直感的な仕組みを説明できる
  • □ BERT（Encoder-only）とGPT（Decoder-only）の違いを説明できる
  • □ ハルシネーションとその対策（RAG等）を説明できる
  • □ Amazon Comprehend / Lex / Translate / Textract / Kendra / Bedrockの用途を区別できる
  • □ Fine-tuningとRAGの使い分けを説明できる

10. 確認クイズ

⏱ 学習時間の目安：15分

理解度確認クイズ（AWS AIF-C01形式）

Q1. ある企業が顧客サポートのメールを自動分類し、緊急度（高・中・低）と担当部署を判定したいと考えています。最も適切なAWSサービスはどれですか？

A. Amazon Kendra
B. Amazon Comprehend カスタム分類
C. Amazon Transcribe
D. Amazon Lex
正解：B — カスタムラベルでのテキスト分類はAmazon Comprehendのカスタム分類機能が適切です。Kendraは検索、Transcribeは音声認識、Lexはチャットボット構築に使います。

Q2. 病院が電子カルテのテキストから患者の診断名・投薬情報・症状を自動抽出したいと考えています。最も適切なサービスはどれですか？

A. Amazon Comprehend
B. Amazon Comprehend Medical
C. Amazon Textract
D. Amazon Bedrock
正解：B — 医療テキスト専用のNLP機能はAmazon Comprehend Medicalです。医療エンティティ（ICD-10コード等）の抽出に特化し、HIPAA対応です。

Q3. LLMが実際には存在しない学術論文を引用してしまう問題の名称として最も適切なものはどれですか？

A. オーバーフィッティング
B. 勾配消失
C. ハルシネーション（幻覚）
D. データドリフト
正解：C — LLMが事実と異なる内容を自信を持って生成する現象をハルシネーション（Hallucination）と呼びます。RAGや出力検証で対策します。

Q4. Transformerアーキテクチャについて正しい記述はどれですか？

A. RNNの一種であり、ゲート機構を改良したモデルである
B. Self-Attention機構を使い、全トークンを並列処理できる
C. 画像認識のためのCNNに基づくアーキテクチャである
D. テキスト生成には使えず、分類タスクのみに対応する
正解：B — TransformerはRNNを使わずSelf-Attentionのみで構成され、全トークンを並列処理します。これにより長距離依存関係の学習と高速な訓練が可能になりました。

Q5. BERTとGPTの最大の違いとして正しいものはどれですか？

A. BERTはテキスト生成に特化し、GPTはテキスト分類に特化している
B. BERTは双方向Transformer（Encoder-only）、GPTは単方向Transformer（Decoder-only）である
C. BERTはGoogle製だが、GPTはAmazon製である
D. BERTは強化学習で学習され、GPTは教師あり学習で学習される
正解：B — BERTは双方向（Encoder-only）で文脈理解・分類タスクに優れ、GPTは単方向（Decoder-only）でテキスト生成に優れます。BERTはGoogle、GPTはOpenAIが開発しました。

Q6. 企業がLLMを導入する際、最新の社内ドキュメントや製品情報を参照して正確な回答を生成したい場合、最も適切なアプローチはどれですか？

A. LLMを大量の社内文書でフルファインチューニングする
B. RAG（Retrieval-Augmented Generation）を実装し、検索した関連文書をプロンプトに追加する
C. プロンプトにすべての社内文書を含めてAPIを呼び出す
D. モデルをゼロから社内データで学習させる
正解：B — RAGは外部知識ベースを検索してプロンプトに追加することで、ファインチューニング不要で最新情報への対応とハルシネーション抑制が可能です。AWS BedrockのKnowledge Basesで実装できます。
