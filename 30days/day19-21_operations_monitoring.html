1. MLOpsの概念とライフサイクル管理

MLOps（Machine Learning Operations）は、機械学習モデルの開発・デプロイ・運用を効率的かつ信頼性高く行うための実践的な手法と文化の総称です。従来のソフトウェア開発のDevOpsをML特有の課題に対応できるよう拡張したものです。

1.1 なぜMLOpsが必要か

従来のソフトウェア開発との違い

  • コードだけでなくデータ・モデル・パラメータも管理対象
  • モデルは時間とともに自然劣化（ドリフト）する
  • 実験の再現性確保が困難
  • データサイエンティストとエンジニアの協働が必要

MLOpsが解決する課題

  • 本番環境への安全・迅速なデプロイ
  • モデル品質の継続的な監視
  • 実験の再現性と追跡可能性
  • スケーラブルなインフラ管理

1.4 モデルレジストリとアーティファクト管理

モデルレジストリとは

モデルレジストリは、学習済みモデルを一元管理するリポジトリです。バージョン管理、メタデータ記録、デプロイ履歴の追跡を行います。


Amazon SageMaker Model Registry の機能：

  • モデルのバージョン管理とグルーピング
  • モデルの承認ステータス管理（Pending → Approved / Rejected）
  • 評価メトリクスとメタデータの保存
  • デプロイパイプラインとの連携
試験ポイント： MLOpsはコード・データ・モデルの3要素を継続的に管理する手法。SageMaker Model Registryはモデルのバージョン管理と承認ワークフローを提供する。

2. モデルデプロイメント戦略

本番環境へのモデルデプロイには複数の戦略があり、リスク管理と性能要件に応じて使い分けます。

2.1 主要なデプロイ戦略

戦略 仕組み メリット デメリット 用途
Blue/Green
デプロイ		 新旧2環境を並行稼働し、一括切替 即時ロールバック可能、ダウンタイムなし コストが2倍 安定性重視の本番環境
カナリア
リリース		 トラフィックの一部（例5-10%）を新モデルへ 実ユーザーで段階的テスト、リスク低減 管理が複雑 大規模サービスの安全な更新
A/Bテスト 複数モデルへトラフィックを分散 実ユーザーでモデル比較が可能 統計的有意差が出るまで時間が必要 モデル性能の実証的比較
シャドウ
デプロイ		 本番トラフィックを新モデルにも複製（結果は非表示） 本番負荷での動作確認、ユーザー影響なし インフラコスト増 リスクゼロでの新モデル検証
ローリング
アップデート		 インスタンスを順次更新 コスト効率が良い 新旧モデルが同時稼働する瞬間あり 一般的な更新シナリオ

2.2 SageMaker エンドポイントのデプロイオプション

リアルタイム推論

低レイテンシ 常時稼働
  • 同期的なリクエスト/レスポンス
  • ミリ秒オーダーの応答が必要なケース
  • エンドポイントは常時起動
  • 例：リアルタイム推薦、詐欺検知

サーバーレス推論

コスト効率 スパイク対応
  • リクエスト時のみ起動（コールドスタートあり）
  • トラフィックが不規則な場合に最適
  • 自動スケーリング
  • 例：バッチ処理、開発環境

非同期推論

大容量ペイロード
  • 最大1GBの入力データ対応
  • 処理完了をSNS通知で受け取り
  • 処理時間が長いタスク向け
  • 例：動画処理、大規模文書解析

バッチ変換

大量データ処理
  • S3データに対するオフライン推論
  • エンドポイント不要（コスト削減）
  • 処理後にインスタンスが自動停止
  • 例：定期レポート生成、データエンリッチメント

2.3 マルチモデルエンドポイントとマルチコンテナエンドポイント

マルチモデルエンドポイント (MME)

単一のエンドポイントで複数のモデルを動的にロードする機能。コスト削減に有効。

  • モデルを必要に応じてメモリへロード/アンロード
  • 数千モデルを1エンドポイントで管理可能
  • テナントごとに異なるモデルを持つSaaSアプリに最適

マルチコンテナエンドポイントとの違い：MMEは同種コンテナで多数のモデルを扱うのに対し、マルチコンテナは異なるフレームワークのコンテナを1エンドポイントに結合する。

試験ポイント： カナリアリリースは段階的なトラフィック移行でリスクを低減。A/Bテストは実ユーザーでモデルを比較評価。シャドウデプロイはユーザーへの影響なしに本番環境でモデルを検証する。

3.4 Amazon SageMaker Clarify との連携

SageMaker ClarifyはModel Monitorと連携してバイアス検出と説明可能性を提供します：

バイアス検出

  • 学習前バイアス：トレーニングデータのバイアス分析

4. Amazon CloudWatch によるMLモニタリング

CloudWatchはAWSの中核的な監視サービスであり、SageMakerエンドポイントのインフラ・パフォーマンス・ビジネスメトリクスを統合監視します。

4.4 Auto Scaling との統合

Application Auto Scaling

SageMakerエンドポイントのインスタンス数を自動調整：

  • ターゲット追跡スケーリング：特定メトリクス値を維持（例：CPUUtilization = 70%）
  • ステップスケーリング：閾値超過時に段階的にスケール
  • スケールアウト/インのクールダウン期間設定

スケーリング設定のベストプラクティス

  • 最小インスタンス数 ≥ 2（高可用性）
  • スケールアウトを速く、スケールインを遅く設定
  • SageMakerVariantInvocationsPerInstanceをスケーリング指標に使用
  • 予測スケーリングでトラフィックパターンに事前対応
試験ポイント： CloudWatchはInvocations、ModelLatency、Invocation5XXErrorsなどSageMakerの標準メトリクスを自動収集。アラームはSNS通知やAuto Scalingと連携して自動対応できる。

5. モデルドリフトの検出と対応

モデルドリフトは、本番環境でのモデル性能が時間とともに低下する現象です。AWSのAI試験では最も重要なトピックの一つです。

5.1 ドリフトの種類

データドリフト（共変量シフト）

入力特徴量の分布が変化するが、入出力の関係は変わらない。


例： EC推薦システムで、コロナ禍により急にホームオフィス製品の検索が急増。モデルはこの新しいパターンに対応できない。

コンセプトドリフト

入出力の関係（学習したパターン）自体が変化する。最も深刻なドリフト。


例： 詐欺検知モデルで、詐欺師が新しい手口を開発。以前は詐欺と関連していた特徴量の意味が変化する。

ラベルドリフト（事前確率シフト）

目的変数（ラベル）の分布が変化する。


例： 二値分類モデルで、ポジティブクラスの割合が学習時10%だったが、本番で30%になった。

予測ドリフト

モデルの予測値の分布が変化する（入力変化の結果）。


例： 価格予測モデルが以前は価格帯全体をカバーしていたが、近年は高価格帯の予測に偏るようになった。

5.2 ドリフト検出の統計的手法

手法 適用対象 仕組み 特徴
Population Stability Index (PSI) 連続値・カテゴリ 分布の変化量を0〜1のスコアで表現 PSI < 0.1: 変化なし、0.1-0.25: 軽微、> 0.25: 重大
KL divergence (KLダイバージェンス) 確率分布 2つの分布の差異を情報理論的に計算 非対称性あり。SageMaker Model Monitorで利用
Wasserstein距離 連続分布 一方の分布をもう一方に変形するコスト 直感的解釈が容易。Earth Mover's Distanceとも
コルモゴロフ・スミルノフ検定 連続分布 累積分布関数の最大差異を検定 非パラメトリック。p値で統計的有意性を評価
カイ二乗検定 カテゴリ変数 観測度数と期待度数の差異を検定 SageMaker Model Monitorのデフォルト検定の一つ

6. CI/CDパイプラインとMLOpsツール

継続的インテグレーション・継続的デリバリー（CI/CD）をMLに適用することで、モデル開発から本番デプロイまでのプロセスを自動化・標準化します。

6.1 Amazon SageMaker Pipelines

SageMaker Pipelines の主要ステップ

SageMaker PipelinesはMLワークフロー全体を宣言的に定義・実行するサービスです。


from sagemaker.workflow.steps import ( ProcessingStep, TrainingStep, CreateModelStep, TransformStep ) from sagemaker.workflow.pipeline import Pipeline from sagemaker.workflow.conditions import ConditionGreaterThanOrEqualTo from sagemaker.workflow.condition_step import ConditionStep # 前処理ステップ step_process = ProcessingStep( name="PreprocessData", processor=sklearn_processor, inputs=[...], outputs=[...] ) # 学習ステップ step_train = TrainingStep( name="TrainModel", estimator=xgb_estimator, inputs={"train": step_process.properties.ProcessingOutputConfig}, depends_on=[step_process] ) # 評価→条件分岐（精度が閾値以上ならデプロイ） condition = ConditionGreaterThanOrEqualTo( left=JsonGet(step_name=step_eval.name, property_file=evaluation_report, json_path="classification_metrics.accuracy.value"), right=0.80 ) step_cond = ConditionStep( name="CheckAccuracyCondition", conditions=[condition], if_steps=[step_register], else_steps=[step_fail] ) pipeline = Pipeline( name="MyMLPipeline", steps=[step_process, step_train, step_eval, step_cond] )

6.3 実験管理 — Amazon SageMaker Experiments

SageMaker Experiments の概要

複数の実験を体系的に管理し、再現性を確保するサービス：

  • 実験（Experiment）：関連する試みのグループ（例：モデル改善プロジェクト）
  • 実行（Run）：個々のトレーニング実行
  • メトリクス記録：損失、精度などの自動ログ
  • パラメータ比較：ハイパーパラメータと結果の可視化比較

import sagemaker from sagemaker.experiments.run import Run # 実験実行のコンテキスト管理 with Run( experiment_name="fraud-detection-v2", run_name="xgboost-lr0.01-depth6", sagemaker_session=sagemaker_session ) as run: run.log_parameter("learning_rate", 0.01) run.log_parameter("max_depth", 6) # 学習実行 model.fit(train_data) run.log_metric("train:accuracy", 0.95) run.log_metric("validation:accuracy", 0.91) run.log_metric("test:auc", 0.93)

6.4 Infrastructure as Code (IaC) による再現性

AWS CloudFormation / AWS CDK の活用：
MLインフラ（エンドポイント設定、監視ルール、IAMロール、VPC設定など）をコードで管理することで、環境間の一貫性を確保し、設定ドリフトを防止します。SageMaker Studio のドメイン設定や実験環境もコード化することが推奨されます。
試験ポイント： SageMaker PipelinesはML特化のワークフローオーケストレーション。条件分岐（ConditionStep）で評価メトリクスが閾値を超えた場合のみデプロイするゲーティングが実装できる。

（量子化・プルーニング）		 推論 インスタンスサイズ縮小 精度とのトレードオフ評価必要

7.2 AWS Inferentia と AWS Trainium

AWS Inferentia (inf1, inf2)

推論専用チップ
  • 大規模推論ワークロード向けAWSカスタムチップ
  • GPU比で高スループット・低コスト
  • AWS Neuron SDKでモデルをコンパイル
  • PyTorch、TensorFlowをサポート
  • LLM推論（Llama、BERT等）に特に有効

AWS Trainium (trn1, trn2)

学習専用チップ
  • 大規模モデルの学習向けAWSカスタムチップ
  • GPU比で最大50%低コスト
  • 同じくAWS Neuron SDKを使用
  • 分散学習に対応（NeuronCore）
  • 基盤モデルの事前学習・ファインチューニングに有効

7.3 責任あるAI (Responsible AI) の運用原則

AWSの責任あるAI フレームワーク

本番環境でのAI運用において、倫理・公平性・透明性を継続的に確保することが求められます。

公平性（Fairness）

  • 保護属性によるバイアスを継続監視
  • SageMaker Clarifyで定期的なバイアス評価
  • 異なるグループ間の性能差を最小化

透明性（Transparency）

  • モデルの判断根拠をステークホルダーに説明
  • SHAP値による予測根拠の提示
  • モデルカード（Model Card）の作成・公開

プライバシー（Privacy）

  • 学習データのPII（個人情報）を適切に処理
  • Amazon Macieでデータのセキュリティ監視
  • 差分プライバシーの適用検討

安全性（Safety）

  • モデルの出力を人間がレビューする仕組み
  • 有害な出力を検出・フィルタリング
  • Amazon Guardrails for Bedrock の活用

頑牢性（Robustness）

  • 敵対的入力への耐性テスト
  • エッジケースでの動作確認
  • 定期的な再評価・再学習

説明責任（Accountability）

  • 全ての決定・変更の監査ログ
  • SageMaker Lineage Trackingで系譜管理
  • 規制要件への準拠記録

7.4 Amazon SageMaker Lineage Tracking

系譜追跡（Lineage Tracking）とは：
モデルのトレーサビリティを確保するため、「このモデルはいつ、どのデータで、どのコードで学習されたか」を自動的に記録します。規制産業（医療・金融）では特に重要で、監査対応や障害時の原因特定に不可欠です。

追跡される情報： データセット → 前処理ジョブ → 学習ジョブ → モデルアーティファクト → エンドポイント

7.5 AWS Cost Explorer と予算アラート

ML コスト管理のベストプラクティス

  • タグ戦略： プロジェクト・チーム・環境別にリソースをタグ付けし、コストを分離
  • AWS Budgets： 月次予算に対する実績・予測でアラートを設定
  • Cost Anomaly Detection： 機械学習を使って異常なコストスパイクを自動検出
  • SageMaker Savings Plans： SageMaker全サービスに適用可能なフレキシブルな割引プラン
試験ポイント： AWS Inferentiaは推論コスト削減、AWS Trainiumは学習コスト削減のための専用チップ。Spotインスタンスは学習コストを最大90%削減できるが、チェックポイントによる耐中断設計が必須。

8. 確認テスト

以下の問題に回答し、3時間の学習内容を確認してください。

練習問題（全10問）

Q1. ある企業が新しいバージョンのMLモデルをデプロイしたいと考えています。本番トラフィックの5%のみを新モデルに送り、問題がなければ段階的に増やす戦略はどれですか？

  1. Blue/Green デプロイメント
  2. カナリアリリース
  3. シャドウデプロイ
  4. ローリングアップデート
回答を表示
正解: B (カナリアリリース)
カナリアリリースは本番トラフィックの一部（少量）を新バージョンへ段階的に移行する戦略です。Blue/Greenは一括切替、シャドウはユーザーへの影響なし（結果を非表示）、ローリングは順次更新です。

Q2. SageMaker Model Monitorには4種類のモニタリングがあります。入力特徴量の統計的分布の変化（平均・標準偏差・欠損率など）を監視するのはどれですか？

  1. モデル品質モニタリング
  2. モデルバイアスモニタリング
  3. データ品質モニタリング
  4. モデル説明可能性モニタリング
回答を表示
正解: C (データ品質モニタリング)
データ品質モニタリングは入力データの統計的特性（平均・標準偏差・欠損率・データ型）を監視します。モデル品質は予測精度（正解ラベルとの比較）、バイアスは公平性、説明可能性はSHAP値の変化を監視します。

Q3. 詐欺検知モデルを運用中に、詐欺師が新しい攻撃手法を開発したため、以前は詐欺と強く相関していた特徴量が意味をなさなくなりました。これはどの種類のドリフトですか？

  1. データドリフト（共変量シフト）
  2. ラベルドリフト（事前確率シフト）
  3. コンセプトドリフト
  4. 予測ドリフト
回答を表示
正解: C (コンセプトドリフト)
コンセプトドリフトは、入出力の関係（学習したパターン）自体が変化する最も深刻なドリフトです。データドリフトは入力分布が変化するもの、ラベルドリフトはラベルの割合が変化するものです。

Q4. SageMaker Pipelinesで、モデルの評価精度が80%以上の場合のみモデルレジストリに登録するゲーティングを実装するために使用するステップタイプはどれですか？

  1. ProcessingStep
  2. TrainingStep
  3. ConditionStep
  4. CreateModelStep
回答を表示
正解: C (ConditionStep)
ConditionStepはメトリクスの条件に基づいてパイプラインの分岐を制御します。ConditionGreaterThanOrEqualToなどの条件を使って「精度≥80%ならif_steps（登録）、それ以外はelse_steps（失敗）」を実装します。

Q5. 推論コストを削減したい企業が、GPU比で大幅にコストを下げられるAWS独自の推論専用チップを使用したいと考えています。どのサービスを使うべきですか？

  1. AWS Trainium
  2. AWS Inferentia
  3. Amazon SageMaker Spot Training
  4. AWS Graviton
回答を表示
正解: B (AWS Inferentia)
AWS Inferentiaは推論専用のAWSカスタムチップで、GPU比で大幅なコスト削減が可能です。AWS Trainiumは学習専用チップ、Spot Trainingは学習コスト削減、Gravitonは汎用ARMプロセッサです。

Q6. MLOps成熟度Level 2の特徴として最も正確なものはどれですか？

  1. すべて手動で実行され、モデルは稀にしか更新されない
  2. MLトレーニングパイプラインは自動化されているが、パイプライン自体のデプロイは手動
  3. MLパイプラインのビルド・テスト・デプロイもCI/CDで完全自動化されている
  4. データ収集のみが自動化されている
回答を表示
正解: C
Level 0: 手動。Level 1: MLトレーニングパイプラインの自動化。Level 2: パイプライン自体のCI/CD（ビルド・テスト・デプロイ）まで完全自動化されている状態です。

Q7. Population Stability Index (PSI) の値が0.30であった場合、どのように解釈しますか？

  1. 変化はほとんどなく、現状のモデルで問題ない
  2. 軽微な変化があるが、モニタリングを継続すれば良い
  3. 重大な変化があり、モデルの再学習を検討すべき
  4. PSIはドリフト検出に使用できない指標である
回答を表示
正解: C
PSIの解釈基準：<0.10 = 変化なし（問題なし）、0.10〜0.25 = 軽微な変化（要監視）、>0.25 = 重大な変化（再学習を検討）。0.30は0.25を超えているため重大な変化と判断します。

Q8. トラフィックが不規則（日中は多く、夜間はほぼゼロ）なアプリケーションで、コストを最小化しながらSageMakerで推論を提供する最適な方法はどれですか？

  1. リアルタイム推論エンドポイント（常時稼働）
  2. サーバーレス推論
  3. バッチ変換ジョブ
  4. 非同期推論エンドポイント
回答を表示
正解: B (サーバーレス推論)
サーバーレス推論はリクエスト時のみ課金され、アイドル時はコストゼロです。トラフィックが不規則でコスト最小化が目標の場合に最適です。ただしコールドスタートによるレイテンシ増加に注意が必要です。

Q9. SageMaker Clarifyが提供する2つの主要機能はどれですか？（2つ選択）

  1. バイアス検出
  2. モデルデプロイメント
  3. 説明可能性（SHAP値）
  4. ハイパーパラメータ最適化
回答を表示
正解: A と C (バイアス検出と説明可能性)
SageMaker Clarifyは、①バイアス検出（学習前・学習後、および本番モニタリング）と②説明可能性（SHAP値による特徴量重要度の計算）を提供します。モデルデプロイはエンドポイント作成、HPOはHyperparameter Tunerが担当します。

Q10. 新しいモデルを本番環境のユーザーへの影響なしに、実際の本番トラフィックを使って検証したい場合、どのデプロイ戦略を使うべきですか？

  1. カナリアリリース
  2. Blue/Green デプロイ
  3. シャドウデプロイ
  4. A/B テスト
回答を表示
正解: C (シャドウデプロイ)
シャドウデプロイは本番トラフィックを新モデルにも複製しますが、新モデルの結果はユーザーへ返却されません（影のように動作）。ユーザーへの影響ゼロで本番負荷での動作を検証できます。カナリアやA/Bはユーザーへの影響があります。

学習まとめ：運用とモニタリングの要点

Day 19-21 で習得した主要概念：
  • MLOps：コード・データ・モデルを統合管理する手法。成熟度はLevel 0〜2で段階的に自動化が進む
  • デプロイ戦略：カナリア（段階的移行）・Blue/Green（一括切替）・シャドウ（影響なし検証）・A/Bテスト（比較実験）の使い分け
  • Model Monitor：データ品質・モデル品質・バイアス・説明可能性の4種類の継続的監視
  • CloudWatch：SageMakerエンドポイントの標準メトリクス収集・アラーム・Auto Scaling連携
  • ドリフト：データドリフト（入力変化）とコンセプトドリフト（関係変化）を区別。PSIで重大度を定量化
  • SageMaker Pipelines：ConditionStepで品質ゲーティングを実装するML特化ワークフロー
  • コスト最適化：Inferentia（推論）・Trainium（学習）でGPUコスト削減。Spot/サーバーレス/Savings Plansの使い分け
  • 責任あるAI：公平性・透明性・プライバシー・安全性・頑牢性・説明責任の6原則を本番で継続維持
