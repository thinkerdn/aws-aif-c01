1. デプロイの基礎概念

学習時間の目安: 25分
このセクションの目標: 「訓練済みモデルを本番で使う」とはどういうことか、推論・エンドポイント・レイテンシ・スループットの意味を理解する。

1.1 デプロイとは何か

機械学習では、訓練（Training）と推論（Inference）は全く異なるフェーズです。訓練はデータからパターンを学習する処理で、多くのGPU・時間・コストを消費します。一方、デプロイとは訓練済みモデルをユーザーや他のシステムが利用できる形で公開することです。

具体例: Amazonの商品レコメンドシステムは、数週間かけて数億件の購買データで訓練されます。しかしユーザーがページを開いたとき、レコメンドは100ミリ秒以内に返す必要があります。このギャップを埋めるのがデプロイ技術です。

1.2 訓練と推論の違い

観点 訓練（Training） 推論（Inference）
目的 モデルのパラメータを最適化 入力に対して予測値を返す
頻度 数日〜数週間に1回 1秒間に数千〜数万回
バッチサイズ 大（数十〜数千件） 小（1〜数件が多い）
ハードウェア 高性能GPU（A100等）が有利 CPUでも十分な場合が多い
重要指標 スループット（時間当たり処理量） レイテンシ（応答速度）
コスト最適化 スポットインスタンスを活用 オートスケーリングが重要

1.3 推論パターン：リアルタイム vs バッチ

リアルタイム推論（Online Inference）

  • リクエストが来るたびに即時応答
  • 低レイテンシが必要（数百ms以内）
  • エンドポイント（常時起動）が必要
  • 例: 不正検知、チャットBot、レコメンド
  • コスト: 常時課金（使用量に関わらず）

バッチ推論（Batch Inference）

  • 大量データをまとめて処理
  • レイテンシよりスループット重視
  • 定期実行（夜間バッチなど）
  • 例: 月次レポート、大規模分類処理
  • コスト: 処理時間のみ課金（安価）

1.4 主要な性能指標

レイテンシ（Latency）: リクエストから応答までの時間（ms単位）
スループット（Throughput）: 単位時間あたりの処理件数（RPS）
可用性（Availability）: システムが稼働している時間の割合（例: 99.9% = 年8.7時間のダウン）
コールドスタート: インスタンスが起動していないときの初回応答遅延
レイテンシとスループットはトレードオフ: バッチサイズを増やすとスループットは上がりますが、1件あたりのレイテンシは上がります。ユースケースに応じて適切なバランスを選択してください。

このセクションの目標: Amazon SageMaker の各エンドポイント種別（リアルタイム・サーバーレス・非同期・バッチ）の違いと使い分けを完全に理解する。

SageMaker のモデルデプロイには 4種類のエンドポイントがあります。ユースケースと要件によって適切な種別を選択します。

2.2 リアルタイムエンドポイント（Real-time Endpoint）

概要と特徴

  • 最もシンプルな常時起動型のエンドポイント
  • インスタンスが常時稼働しているため、即座に応答可能
  • コスト: インスタンス稼働時間に応じて課金
  • オートスケーリング設定が可能（負荷に応じてインスタンス追加）
  • ペイロード上限: 6MB、タイムアウト: 60秒
# SageMaker リアルタイムエンドポイントの作成例（Python SDK） import boto3 import sagemaker from sagemaker.sklearn.model import SKLearnModel # モデルオブジェクトの作成 model = SKLearnModel( model_data="s3://my-bucket/model.tar.gz", role="arn:aws:iam::123456789:role/SageMakerRole", entry_point="inference.py", framework_version="1.2-1" ) # エンドポイントへデプロイ predictor = model.deploy( initial_instance_count=1, instance_type="ml.m5.large", endpoint_name="my-realtime-endpoint" ) # 推論の呼び出し result = predictor.predict([[5.1, 3.5, 1.4, 0.2]]) print(result) # → [0] (アイリスの品種クラス)

2.3 サーバーレスエンドポイント（Serverless Inference）

概要と特徴

  • 使用時のみ課金（リクエストがないときは課金ゼロ）
  • インスタンス管理が不要（自動スケーリング）
  • コールドスタート（初回起動遅延）が発生する（数秒程度）
  • 低頻度のリクエストに最適
  • メモリ設定: 1GB〜6GB、タイムアウト: 最大60秒
# サーバーレスエンドポイントの設定例 from sagemaker.serverless import ServerlessInferenceConfig serverless_config = ServerlessInferenceConfig( memory_size_in_mb=2048, # 2GB メモリ max_concurrency=10, # 同時実行数の上限 provisioned_concurrency=1 # コールドスタート対策（オプション） ) predictor = model.deploy( serverless_inference_config=serverless_config, endpoint_name="my-serverless-endpoint" )

2.4 非同期エンドポイント（Asynchronous Inference）

概要と特徴

  • 長時間かかる推論（数分〜）に最適
  • リクエストをキューに積み、非同期で処理
  • 処理完了後に S3 に結果を保存、SNS通知も可能
  • アイドル時にインスタンス数をゼロにスケールダウン可能
  • ペイロード上限: 1GB（大きなファイルも処理可能）
非同期推論のフロー:

クライアント            SageMaker               S3
    |                      |                    |
    |-- リクエスト送信 -->  |                    |
    |                      |-- キューに積む      |
    |<-- RequestID 即返却 --|                    |
    |                      |-- 推論処理中...     |
    |  （他の処理を継続）   |                    |
    |                      |-- 完了 ----------->|（結果保存）
    |                      |-- SNS通知          |
    |  （通知受け取り）     |                    |
    |-- 結果取得リクエスト --------------------->|
    |<-- 予測結果 -------------------------------|

2.5 バッチ変換（Batch Transform）

概要と特徴

  • S3 のデータをまとめて処理し、結果を S3 に書き戻す
  • エンドポイントを常時稼働させる必要がなく、処理時間のみ課金
  • 大規模なオフライン推論に最適（数百万件など）
  • 処理完了後はインスタンスが自動的に終了
  • AssembleWith オプションで出力の結合方法を制御
# バッチ変換ジョブの実行例 import sagemaker transformer = model.transformer( instance_count=2, instance_type="ml.m5.xlarge", output_path="s3://my-bucket/batch-output/", assemble_with="Line", # 出力を行単位で結合 accept="text/csv" ) transformer.transform( data="s3://my-bucket/batch-input/data.csv", content_type="text/csv", split_type="Line" # 入力を行単位で分割して並列処理 ) transformer.wait() print("バッチ変換完了")

2.6 エンドポイント種別の選択ガイド

エンドポイント種別 レイテンシ コスト 最適なユースケース
リアルタイム 低（常時起動） 常時課金 高頻度・低レイテンシが必要
例: ECサイトのリアルタイム推薦
サーバーレス コールドスタートあり リクエスト時のみ 低頻度リクエスト
例: 社内ツール、開発環境
非同期 高（非同期処理） アイドル時ゼロ可 長時間推論・大ペイロード
例: 動画解析、大量文書処理
バッチ変換 該当なし（オフライン） 処理時間のみ 大規模オフラインバッチ
例: 月次スコアリング
AWS試験ポイント: 「コストを抑えたい」→ サーバーレス or バッチ変換。「リアルタイム性が必要」→ リアルタイムエンドポイント。「処理時間が長い・ファイルサイズが大きい」→ 非同期推論。この選択基準が頻出です。

3. デプロイ戦略（ブルー/グリーン・カナリア・A/Bテスト）

学習時間の目安: 30分
このセクションの目標: 本番環境でモデルを安全に更新するための3つの戦略を理解し、SageMakerでの実装方法を把握する。

3.1 なぜデプロイ戦略が重要か

新しいモデルを本番環境に直接デプロイするのは危険です。新モデルが予期せぬバグや性能劣化を持っていても、すべてのユーザーに即座に影響します。安全なデプロイ戦略は、リスクを段階的に管理するための仕組みです。

直接切り替えの危険性: モデルv1からv2に直接全切り替えした場合、v2に問題（精度低下、クラッシュ、異常な予測）があると、全ユーザーに影響が出るまで問題が検知されません。ロールバックにも時間がかかります。

3.2 ブルー/グリーンデプロイ

ブルー/グリーンデプロイ:

フェーズ1（切り替え前）:
  トラフィック 100%  -->  [Blue環境: モデルv1]
                          [Green環境: モデルv2]  <-- テスト中（本番トラフィックなし）

フェーズ2（検証後に瞬時切り替え）:
  トラフィック 100%  -->  [Green環境: モデルv2]  <-- 本番稼働
                          [Blue環境: モデルv1]   <-- 待機（すぐロールバック可）

フェーズ3（安定確認後）:
  Blue環境を削除してコスト削減
メリット:
  • 問題発生時に即座にロールバック可能
  • Green環境で本番前テストが可能
  • ダウンタイムなしで切り替え
  • SageMaker が標準サポート
デメリット:
  • 切り替え中は2環境分のコストが発生
  • 完全移行まで旧モデルが稼働し続ける
  • 問題が小規模に発現しない

3.3 カナリアデプロイ

カナリアデプロイ（段階的なトラフィック移行）:

フェーズ1: 新モデルに少量のトラフィックを流す
  トラフィック: 95% --> [モデルv1（既存）]
                5% --> [モデルv2（新）]   <-- メトリクス監視

フェーズ2: 問題なければ比率を増やす
  トラフィック: 70% --> [モデルv1]
               30% --> [モデルv2]

フェーズ3: 完全移行
  トラフィック:  0% --> [モデルv1]  <-- 削除
              100% --> [モデルv2]

問題検知時: 即座に0%に戻してロールバック

カナリアという名前の由来

炭鉱で有毒ガスを早期検知するためにカナリアを持ち込んでいた習慣から。新モデルを少数のユーザーにだけ公開し、問題が大きくなる前に検知するという考え方です。

# SageMaker カナリアデプロイ設定例 import boto3 sm_client = boto3.client("sagemaker") # トラフィック分割設定（新モデルに10%のみ） sm_client.update_endpoint( EndpointName="my-endpoint", EndpointConfigName="my-endpoint-config-v2", DeploymentConfig={ "BlueGreenUpdatePolicy": { "TrafficRoutingConfiguration": { "Type": "CANARY", "CanarySize": { "Type": "CAPACITY_PERCENT", "Value": 10 # 最初は10%のみ新モデルへ }, "WaitIntervalInSeconds": 300 # 5分監視後に残り移行 }, "TerminationWaitInSeconds": 600 } } )

3.4 A/Bテスト

A/Bテストはカナリアデプロイと似ていますが、どちらのモデルが優れているかを実験的に検証する目的で使います。ビジネスメトリクス（クリック率、購買率）で比較評価します。

A/Bテスト（モデル性能の比較実験）:

  ユーザーA群（50%）  -->  [モデルA（現行）]  --> 購買率: 3.2%
  ユーザーB群（50%）  -->  [モデルB（新規）]  --> 購買率: 3.8%

  結果: モデルB が統計的に有意に優れている → 完全移行を決定

  ランダムに割り当て（クッキーなどで同一ユーザーは同じモデルへ）
  十分なサンプル数が集まるまで実験継続（統計的検定が必要）
戦略 主な目的 リスク管理 SageMakerサポート
ブルー/グリーン ゼロダウンタイムでの切り替え 問題時に即ロールバック 標準機能（update_endpoint）
カナリア 段階的な移行でリスク低減 小規模で問題を早期検知 BlueGreenUpdatePolicy
A/Bテスト どちらが優れるか実験で検証 実験規模でリスクを制御 ProductionVariants で実装
AWS試験ポイント: SageMaker の ProductionVariants を使うと、1つのエンドポイントに複数のモデルバリアントを設定し、各バリアントへのトラフィック比率を指定できます。A/BテストやカナリアデプロイはこのProductionVariantsで実現します。

4. コンテナ化とサーバーレスデプロイ

学習時間の目安: 25分
このセクションの目標: Dockerコンテナを使ったモデルのパッケージング、AWS Lambda・ECS・EKSへのデプロイ方法を理解する。

4.1 なぜコンテナが必要か

機械学習モデルの推論には特定のフレームワーク（TensorFlow, PyTorch）やライブラリのバージョンが必要です。コンテナは「依存関係ごとモデルをパッケージング」することで、環境の差異によるバグを防ぎます。

「自分のPCでは動くのに本番で動かない」問題:
開発環境: Python 3.9 + PyTorch 2.0 + CUDA 11.8
本番環境: Python 3.8 + PyTorch 1.13 + CUDA 11.6
→ Dockerコンテナを使えばこの問題が解消します（環境ごとパッケージ化）

4.2 モデルサービング用Dockerfileの例

# Dockerfile（推論サーバー用コンテナの例） FROM python:3.11-slim # 依存ライブラリのインストール WORKDIR /app COPY requirements.txt . RUN pip install --no-cache-dir -r requirements.txt # モデルと推論コードのコピー COPY model/ /app/model/ COPY inference.py /app/ # ヘルスチェック用エンドポイント EXPOSE 8080 CMD ["python", "inference.py"] ## requirements.txt の内容例 ## # fastapi==0.104.0 # uvicorn==0.24.0 # scikit-learn==1.3.2 # numpy==1.24.3

4.3 AWS Lambda を使ったサーバーレス推論

Lambda + コンテナイメージによる推論

AWS Lambda はコンテナイメージ（最大10GB）でのデプロイをサポートしており、軽量なMLモデルのサーバーレス推論に使えます。

# Lambda 推論関数の例（scikit-learnモデル） import json import pickle import numpy as np # Lambdaの初期化時にモデルをロード（コールドスタートで1回だけ実行） with open("/var/task/model.pkl", "rb") as f: model = pickle.load(f) def handler(event, context): """Lambda のメインハンドラー""" body = json.loads(event["body"]) features = np.array(body["features"]).reshape(1, -1) prediction = model.predict(features) probability = model.predict_proba(features) return { "statusCode": 200, "body": json.dumps({ "prediction": int(prediction[0]), "probability": probability[0].tolist() }) }

5. エッジデプロイと推論最適化

学習時間の目安: 25分
このセクションの目標: クラウドではなくエッジデバイスにモデルをデプロイするシナリオと、推論を高速化するための量子化・プルーニング・SageMaker Neo等の手法を理解する。

5.1 エッジ推論とは

通常のML推論はクラウド（AWSサーバー）で行いますが、エッジデプロイではモデルをデバイス（工場のカメラ、スマートフォン、IoTセンサー）に直接配置して推論します。

エッジ推論が必要な場面

  • ネットワークが使えない環境（工場、農場）
  • 超低レイテンシが必要（自動運転: 1ms以下）
  • プライバシー要件（データをクラウドに送れない）
  • 通信コストの削減（大量の画像をクラウドに送るのは高コスト）

エッジ推論の課題

  • デバイスのメモリ・CPU・バッテリーが限られる
  • モデルを小さく・軽くする必要がある
  • ハードウェアごとの最適化が必要
  • モデルの更新・管理が複雑

5.2 AWS IoT Greengrass と SageMaker Edge Manager

エッジデプロイのアーキテクチャ（AWSサービス構成）:

[AWSクラウド]
  SageMaker ----> 訓練 ----> モデル
  Neo Compiler --> 最適化 --> 軽量モデル
  S3 -----> モデル保管
  IoT Core --> デバイス管理
       |
       | MQTT / TLS
       |
[エッジデバイス]
  IoT Greengrass ----> コンポーネント管理
  SageMaker Edge Agent --> 推論実行
  ローカルモデル --------> オフライン推論可能

例: 工場カメラ -> Greengrass -> 製品不良検知モデル -> 不良品ライン停止
AWS SageMaker Neo: モデルを特定のターゲットハードウェア（Raspberry Pi、Intel、ARMなど）向けにコンパイル・最適化するサービスです。最大25倍の推論速度向上を実現する場合があります。

5.4 量子化の詳細

FP32（32ビット浮動小数点）: 1つの重みに 4バイト使用
INT8（8ビット整数）: 1つの重みに 1バイト使用 → サイズ 75% 削減
FP16（16ビット浮動小数点）: 1つの重みに 2バイト使用 → サイズ 50% 削減

GPT-3（1750億パラメータ）の例:
FP32: 700GB → INT8: 175GB → INT4: 87GB
量子化の注意: 量子化後は元のモデルとわずかに異なる予測をすることがあります。本番適用前に量子化後モデルの精度を必ず評価してください。Post-Training Quantization（訓練後量子化）が最もシンプルな手法です。

6. モデルの監視と更新（MLOps）

学習時間の目安: 25分
このセクションの目標: デプロイ後のモデルを継続的に監視し、性能劣化時に再訓練・再デプロイする MLOps の考え方とAWSでの実装を理解する。

6.1 デプロイ後の監視が必要な理由

モデルは一度デプロイすれば永遠に使えるわけではありません。世界は変化し、入力データの分布も変わります。監視なしでは、モデルの性能が静かに劣化していくことに気づけません。

現実の例: コロナ禍前に訓練されたホテル需要予測モデルは、2020年3月以降に急激に精度が落ちました。「旅行」「宿泊」「出張」などの単語や行動パターンが、過去のデータとは全く異なる意味を持つようになったためです（コンセプトドリフト）。

6.2 SageMaker Model Monitor

Model Monitor の4種類の監視

監視タイプ 検知対象 具体例
Data Quality 入力データの統計的分布の変化 年齢の平均値が35→25に変化
Model Quality 予測精度の変化（正解ラベルが必要） Accuracy が90%→75%に低下
Bias Drift 公平性メトリクスの変化 特定の性別・人種への予測バイアス
Feature Attribution Drift SHAP値（特徴量重要度）の変化 重要だった特徴量の影響度が低下

6.4 SageMaker Pipelines（MLパイプラインの自動化）

SageMaker Pipelines とは

データ前処理 → 訓練 → 評価 → モデル登録 → デプロイ の一連のステップをコードとして定義・自動実行するサービスです。GitにコードをPushするだけでMLパイプラインが自動実行されるCI/CD（継続的インテグレーション/デリバリー）を実現します。

# SageMaker Pipelines の定義例（概略） from sagemaker.workflow.pipeline import Pipeline from sagemaker.workflow.steps import ProcessingStep, TrainingStep from sagemaker.workflow.conditions import ConditionGreaterThanOrEqualTo from sagemaker.workflow.condition_step import ConditionStep # 前処理ステップ step_process = ProcessingStep(name="Preprocessing", ...) # 訓練ステップ（前処理の完了後に実行） step_train = TrainingStep(name="Training", ...) # 評価後の条件分岐（精度が85%以上なら登録、未満なら失敗） cond_gte = ConditionGreaterThanOrEqualTo( left=JsonGet(step_name=step_eval.name, property_file=..., json_path="accuracy"), right=0.85 ) step_cond = ConditionStep( name="AccuracyCheck", conditions=[cond_gte], if_steps=[step_register], # 条件満足 → モデル登録 else_steps=[step_fail] # 条件不満 → パイプライン失敗 ) # パイプラインの定義と実行 pipeline = Pipeline( name="MLPipeline", steps=[step_process, step_train, step_eval, step_cond] ) pipeline.upsert(role_arn=role) pipeline.start()
AWS試験ポイント:
  • SageMaker Pipelines: MLワークフローの自動化・バージョン管理
  • SageMaker Model Registry: モデルのバージョン管理・承認フロー
  • SageMaker Model Monitor: 本番モデルのドリフト検知・品質監視
  • SageMaker Feature Store: 特徴量の一元管理・再利用

演習問題（AIF-C01 スタイル）

問1. ある企業が毎月末に100万件の顧客データに対してスコアリングを行う必要があります。最もコスト効率の良いデプロイ方法はどれですか？

A) リアルタイムエンドポイント（常時稼働）
B) サーバーレスエンドポイント
C) バッチ変換（Batch Transform） ★正解★
D) 非同期エンドポイント

解説: 月次の大量バッチ処理はバッチ変換が最適。S3→推論→S3のフローで、処理中のみ課金。リアルタイムは常時課金で無駄が多い。

問2. 工場のカメラで撮影した製品画像をインターネット接続なしでリアルタイムに不良品判定したい場合、最適なAWSアーキテクチャはどれですか？

A) SageMaker リアルタイムエンドポイント + API Gateway
B) AWS IoT Greengrass + SageMaker Neo でエッジデプロイ ★正解★
C) AWS Lambda + S3
D) Amazon EKS + Fargate

解説: インターネット接続なし → エッジデプロイが必須。Greengrass がエッジでのコンポーネント管理を担い、Neo がデバイス向け最適化を行う。

問3. 本番環境で稼働中のモデルv1を新しいモデルv2に安全に更新したいです。最初はトラフィックの5%だけv2に流し、問題がなければ段階的に増やしたい場合、SageMakerの何を使いますか？

A) SageMaker Batch Transform
B) SageMaker Model Monitor
C) BlueGreenUpdatePolicy（カナリアデプロイ） ★正解★
D) SageMaker Pipelines

解説: 段階的なトラフィック移行 = カナリアデプロイ。SageMakerではBlueGreenUpdatePolicyのTYPE="CANARY"で設定。ProductionVariantsでも実現可能。

問4. 動画ファイル（1ファイル500MB）に対して感情分析を行うMLシステムを構築します。処理は非同期で構わず、完了したら通知を受け取りたいです。最適なSageMakerのデプロイ方法はどれですか？

A) リアルタイムエンドポイント（ペイロード上限6MB）
B) サーバーレスエンドポイント（メモリ上限6GB）
C) 非同期エンドポイント + SNS通知 ★正解★
D) AWS Lambda（デプロイパッケージ上限250MB）

解説: 500MBの大ペイロード + 非同期処理 + 完了通知 → 非同期エンドポイント（ペイロード上限1GB）+ SNS通知が最適。リアルタイムはペイロード上限6MBで不可。
